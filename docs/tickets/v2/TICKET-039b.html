Requirements

  1. The List tab renders a paper-warm card with a sticky column header and a scrollable body. Plant rows are grouped by space, in the order spaces appear in useSpaces.
  2. Each plant row shows avatar (coral urgent ring on overdue), name, species (Fraunces italic), next-care label, and a mood dot (thriving / thirsty / wilting).
  3. The mood dot animates subtly (mood-breathe for thriving + thirsty, mood-pulse for wilting). Users with prefers-reduced-motion: reduce get a static dot.
  4. Tapping a plant row navigates to /plants/:id. The link works even if Plant Detail is a placeholder page.
  5. Each space group has a header — Fraunces italic + space icon + name + plant count. Outdoor groups also show a weather pill (sky or frost) when weather data is loaded; otherwise the pill is omitted.
  6. The first body row is an "Add a space" CTA. Clicking opens the same AddCustomSpaceForm dialog used by the Rooms grid; saving creates a new space + closes the dialog + a new (empty) space group appears at the bottom.
  7. Tapping a RoomCard in the Rooms tab jumps to List view filtered by that space, and a "Filtered by {name}" chip appears at the top with a clear-X. Clearing restores the full grouped list.
  8. The search input filters plant rows by nickname OR species common name (case-insensitive contains). Empty groups (no matching plants) hide while search is active. Clearing search restores the full grouped list.
  9. If the user has zero plants, the List view shows an empty state with a CTA routing to /add-plant.
  10. Backend stays untouched — same useSpaces + usePlants + useWeather data already loaded by the page.

React concepts worth looking up

Group-by + filter pipelines in useMemo. The list body needs three derivations: filter plants by space (when chip set), filter by search query, group by space. Naively re-computing on every render — including unrelated state changes like dialog open/close — is wasted work. Wrap in useMemo with deps [plants, spaces, searchQuery, filteredSpaceId] so it only recomputes when one of those four changes. Same pattern Rails would use with a memoized helper method on a presenter.
useDeferredValue for search input. React 18 added a hook that lets the input render at full responsiveness while the heavy downstream work (filter + regroup) catches up at lower priority. Useful when the list is genuinely expensive to recompute. With ≤30 plants you almost certainly don't need it — measure first. If the input feels laggy under typing, reach for it; otherwise skip. (Project precedent: Step 3's species search uses useDebouncedValue for a similar reason; both work, pick one.)
URL search-params vs local state for the filter chip. Two options: store filteredSpaceId in useState, or push it into the URL via useSearchParams from react-router-dom (?view=list&space_id=N). URL state survives refresh + makes "deep-link to filtered list" trivial. Local state is simpler. Pick one + commit — don't half-implement both. Recommendation: URL for both view and space_id so refreshing keeps the user where they were. Rails analog: params[:view] / params[:space_id].
Prefers-reduced-motion via CSS media query. The mood dot animations live in globals.css as @utility blocks. Wrap the animation rules in @media (prefers-reduced-motion: reduce) { animation: none; } so OS-level "reduce motion" settings auto-disable them. No JS needed for this case (unlike DayRituals' Framer drag, which uses useReducedMotion because Framer doesn't auto-respect the media query).
Compound component vs plain props for PlantRow. CLAUDE.md's slot decision tree: single-paired slots = props (Rails' renders_one), multi-occurrence = compound children (renders_many). PlantRow has fixed slots (avatar, title block, next-care, mood dot) — plain prop API. Don't reach for <PlantRow><Avatar /><Title />…</PlantRow> here; it's overkill.

Notes & gotchas

  • Reuse the AddCustomSpaceForm dialog. R3a's House.jsx already holds dialogState + handlers. The list's "Add a space" row opens the same dialog. Don't duplicate state.
  • Plant Detail (/plants/:id) is R4 territory. Row clicks navigate cleanly; the target may 404 until TICKET-040 ships. That's acceptable — don't try to fix the destination here.
  • Mood-dot keyframes go in globals.css as @utility blocks. CLAUDE.md's CSS rule: no inline style={{ animation: … }} for decorative CSS. Match the hero-glow-urgent / toast-progress precedent.
  • Filter-chip state: pick URL OR local, not both. URL via useSearchParams survives refresh + deep-links nicely; local state is simpler. Recommendation: URL. R3a left the chip stubbed — wire whichever pattern you pick consistently.
  • Don't half-implement R3c. Search input + filter chip + plant rows are R3b. The mobile floating drawer that overlays MobileTopBar with the search input is R3c — a separate ticket.
