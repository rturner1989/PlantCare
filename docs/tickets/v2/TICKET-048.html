TICKET-048 · R11 Follow-up

Care anchor on plant create

Open High

Summary

Add Plant currently leaves last_watered_at + last_fed_at empty when a plant is created. The schedule has no real anchor, so mature plants joining the platform get silently-wrong "next water in N days" until the user logs a real care action. This ticket adds mandatory date pickers to the Add Plant flow (cross-cutting dialog + onboarding parity) and the backend validation to back them up.

In one sentence: every new plant tells the system when it was last watered and fed, so the schedule starts in the right place.

Requirements

  1. Adding a new plant requires the user to pick when it was last watered. Default value is today's date so the common case (just-bought plant) is one tap to submit.
  2. If the picked species has a feeding cycle, the user also picks when it was last fed. Same default-today behaviour. If the species has no feeding cycle, no feed picker renders and the plant saves with last_fed_at nil.
  3. The server rejects future-dated anchors with a 422 + field-level validation error. The client surfaces the error inline on the appropriate picker via the existing useFormSubmit + ValidationError pattern.
  4. The server rejects anchors older than 12 months with the same 422 + field error pattern.
  5. Onboarding's plant-add path (PlantFormDialog inside Step3Plants) shows the same pickers with the same defaults — no shortcut for first-run users.
  6. Existing plants are unaffected — the migration / model change only validates writes; reads stay backward-compatible. (No migration needed; columns already exist.)

Acceptance criteria

Open the Add Plant dialog from Today's empty state. After picking a species, the details step shows a "When did you last water?" date input pre-filled with today's date. If the species feeds, a second "When did you last feed?" picker also shows.
Pick a species with no feeding cycle (e.g. Snake Plant if seeded as non-feeding, or check via DevTools that species.feeding_frequency_days is null). Only the watering picker renders.
Try to set the watering date to tomorrow → server returns 422, picker shows red border + inline error, focus returns to the picker.
Try to set the watering date to 2 years ago → same 422 + inline error pattern.
Submit a valid plant with last-watered = today, last-fed = 3 days ago. Open Plant Detail → "Watered today" and "Last fed 3 days ago" read correctly. Next-water countdown calculates from today, not from the absent anchor.
In onboarding Step 3, picking a species + filling nickname now also requires the date pickers. Going Back → Forward through the wizard preserves the picked dates.
DevTools → Network on submit: the POST /api/v1/plants payload includes plant[last_watered_at] and (when applicable) plant[last_fed_at] as ISO 8601 strings.
Lint and tests pass.

React concepts worth looking up

Native form controls vs custom widgets. <input type="date"> uses each browser's native date picker — works on iOS, Android, Chrome, Safari, Firefox without any library. The downside is you can't restyle the picker UI. We trade visual control for zero JS, full keyboard support, and consistent OS UX. If you find yourself reaching for react-datepicker or similar, ask whether the visual win is worth the bundle cost first.
Date strings vs Date objects in JSON payloads. Rails' datetime columns parse ISO 8601 strings ("2026-05-08" or "2026-05-08T00:00:00Z") automatically. The native date input gives you "YYYY-MM-DD" on its value. Don't bother with new Date() + toISOString() — pass the string straight through. Less code, no timezone ambiguity from JS implicit conversions.
useFormSubmit + ValidationError, again. Same pattern StepDetails already uses — throw new ValidationError({ last_watered_at: 'Date is in the future' }) from the action, the hook routes it into fieldErrors, the picker reads error={fieldErrors.last_watered_at}. No new mechanism. Server-side 422 lands in the same shape (the api client translates Rails' field-keyed errors).
Conditional rendering driven by data shape, not by user role. The feeding picker shows when species.feeding_frequency_days is present. Don't introduce a feature flag or "edit mode" prop — the data tells you whether the question applies. Same shape as how StepDetails already conditionally hides the space picker when defaultSpaceId is set.

Notes & gotchas

  • Don't bundle TICKET-049 into this. 049 is the overdue visual sweep — different concern (input vs output), different review shape (frontend-only). Stay scoped.
  • Don't touch adaptive scheduling. Phase 2 territory. This ticket adds the anchor; it doesn't learn from user behaviour.
  • Watch out for timezone slippage. The native picker emits a date string in the user's local calendar. Rails parses it as midnight UTC. For "last watered yesterday" semantics this is fine; if you ever surface "X hours ago" you'll need to think harder about TZs.
  • Don't introduce a Date object in the payload. Pass the string straight through to the mutation. Date objects + JSON.stringify produce ISO timestamps with implicit UTC midnight, which is what you want — but mixing string vs Date in the same field causes confusion downstream.
  • onboarding's submit lifecycle is split between PlantFormDialog and Step3Plants. The dialog owns the pickers + validation; Step3Plants's handleConfirmAdd stays the species hydration + create call. Don't re-route mutation logic through the dialog.
  • Run the review skills before PR. /vercel-react-best-practices + /accessibility (date inputs need label association — DateInput should handle it). DHH-style review for the backend params + validation change.
