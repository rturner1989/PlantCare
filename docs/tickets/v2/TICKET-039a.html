Requirements

  1. The Rooms tab renders one card per user space, in a responsive grid (2 cols mobile, 3 cols desktop), styled per mockup 21.
  2. Each room card shows the space's icon (emoji), name, plant count, and an indoor/outdoor flag.
  3. Each room card shows up to 4 plant peek avatars; if the space has 5+ plants, the 4th cell shows "+N more".
  4. Each room card shows a "next-due plant" summary row, derived client-side from the most-overdue (or soonest-due) plant in that space.
  5. Indoor rooms show a second summary row with the space's light + humidity ("Bright · average humidity"). Outdoor rooms show a weather pill instead, using useWeather; if weather data is unavailable, fall back to the light/humidity row.
  6. An attention badge in coral appears on the card when ≥1 plant in the space is overdue. The page header's "N needs attention" line aggregates these counts.
  7. The first cell in the Rooms grid is an Add-a-space tile that opens AddSpaceDialog. Saving the dialog adds the new space to the grid (TanStack Query cache invalidates).
  8. When the user has zero spaces, the page shows an empty state with a CTA routing to /welcome.
  9. Tapping a room card jumps to List view filtered by that space — but the List tab is a placeholder until R3b. Acceptable interim: store the selected space ID in local state or URL param so R3b can pick it up.
  10. Backend stays untouched. All derivation (peek, next-due, attention count, indoor/outdoor) is client-side off the existing usePlants() + useSpaces() data.

Acceptance criteria

Visiting /house on a user with seeded spaces shows the Rooms grid with mockup-21 styling — paper-warm header card, italic display heading, view toggle.
Each room card displays icon + name + plant count + indoor/outdoor chip + summary rows + (when applicable) attention badge.
A room with 5+ plants shows 3 plant avatars + a "+N more" tile in the peek strip.
An outdoor room (e.g. Patio) shows a weather pill summary row when weather data is loaded; falls back to light/humidity when not.
A room with one or more overdue plants shows the coral attention badge; the page header's "N needs attention" matches the sum across rooms.
Tapping the Add-a-space tile opens a dialog identical (or near-identical) to the one used in R9 onboarding. Submitting saves a new space + closes the dialog + the new card appears in the grid.
On a fresh user with zero spaces, the page renders an empty state with a CTA that routes to /welcome.
The page renders correctly at mobile (375px), tablet (768px), and desktop (1280px) breakpoints — grid columns adapt, no overflow.
The List tab is either visibly disabled, or renders a "Coming in R3b" placeholder. No half-broken List view.
Lint passes (./scripts/lint.sh); Vitest passes (npx vitest run); 1+ Playwright spec covers the Rooms grid happy path + Add-a-space dialog open.

React concepts worth looking up

Compound components for slot-based primitives. RoomCard has multiple distinct slots (icon, title block, peek strip, summary rows, attention badge). The project pattern (per CLAUDE.md) is: single-paired slots become props (renders_one in Rails), multi-occurrence slots become compound children (renders_many). For RoomCard, the icon + title + badge are paired-slot props; the summary rows are multi-occurrence children. Card primitive already exemplifies this with Card.Header / Card.Body / Card.Footer.
useMemo for per-space derivations. Computing peek + next-due + attention naively means iterating plants × spaces on every render. Wrap the per-space derivation in useMemo([plants, spaces]) so it only recomputes when one of those changes — the rest of the page (header counts, segment toggle state) re-renders without recomputing the heavy derivation. Rails analog: a memoized helper method on a presenter that closes over @plants.
TanStack Query cache invalidation pattern. When the Add-a-space dialog saves, the new space needs to appear in the grid without a manual refetch. The project pattern (per CLAUDE.md Mutation cache section) is invalidateQueries(['spaces']) after the mutation succeeds — TanStack refetches the index query and the grid updates. Don't reach for setQueriesData here; the Add-a-space mutation has cascading effects (counter caches on User), so plain invalidate is correct.
Conditional rendering: variant via class modifier vs separate component. Indoor and outdoor rooms differ only in their second summary row. Two options: (a) one RoomCard with a prop or className modifier that swaps the row internally, (b) two separate components (IndoorRoomCard + OutdoorRoomCard). For one-line difference, pick (a) — keeps the API surface small. If R3b later adds more variant divergence, revisit. Composition lesson, not a hard rule.
Empty-state pattern. When spaces.length === 0, render EmptyState with the icon + title + description + action props. Don't conditionally render the grid as spaces.length > 0 ? grid : empty with the grid's chrome above the empty state — that creates the awkward "header without content" UX. The empty state replaces the grid entirely (under the same header).

Notes & gotchas

  • Don't touch List view yet. Leave the segment toggle wired so clicking "List" doesn't crash, but render a placeholder div / EmptyState there. Plant tiles + search + filter chip ship in R3b.
  • AddCustomSpaceForm is currently under onboarding. Either move it to components/spaces/ and re-export from the onboarding path, or duplicate temporarily and consolidate in R3b. Onboarding flow must keep working.
  • Weather pill needs graceful fallback. useWeather may return null when the user hasn't granted location yet. Outdoor cards must fall back to the light/humidity row in that case — never render an empty pill.
  • Space.icon is a string slug, not an emoji. 'couch' / 'bed' / 'kitchen' / 'patio' / … need mapping to emoji. components/onboarding/Step2Spaces.jsx already has an ICONS const — extract it to utils/spaceIcons.js if RoomCard needs it too.
  • Filter-by-space carry-over to R3b. Tapping a card jumps to List view. Until R3b lands, store the selected space ID either in local state, URL param (?space=N), or simply log a TODO and have the click do nothing visible yet. Pick whichever feels cleanest at build time.
