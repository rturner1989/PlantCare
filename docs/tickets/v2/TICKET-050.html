Toast redesign — glass vessel + bold chip + structured payload

Open High

Summary

Replace the current TailwindUI-stock toast with the locked v2 design (mockup 31): a translucent glass vessel + bold gradient chip + structured title/meta/action payload. Adds two new kinds (undo, loading) and a resolve flow so loading toasts can transition into success/error in place. Existing call sites (toast.success('msg')) keep working — single-string remains valid.

In one sentence: repaint the toast to v2, give it title/meta/action slots, and let loading toasts swap in a result instead of stacking a second toast.

Mental model: Rails has flash[:notice] as a single one-shot string-keyed message bag. We're upgrading our equivalent from "one string per type" to a structured payload (title + meta + action) and adding a "promise/resolve" pair so async ops can show a spinner toast then swap to success/error in place.

Requirements

  1. Glass vessel. Toast renders as a translucent paper card with backdrop-blur and an inset white highlight — the same recipe as the notifications drawer, scaled down. Sits cleanly on top of the mint-gradient app background without colour shift.
  2. Bold chip per kind. A 36px circle filled with a gradient unique to the kind (success emerald→leaf, error coral→coral-deep, warn sunshine→sunshine-deep, info sky→sky-deep, undo mint→leaf, loading paper-deep with spinner ring). Chip carries the icon/emoji; vessel is colour-neutral.
  3. Structured payload. Toasts have a bold title, optional Fraunces-italic meta line, and optional action button. Title can include a category-coloured <em> highlight (e.g. Monty watered).
  4. Six kinds. success / undo / error / warn / info / loading — callable as toast.success(...), toast.undo(...), etc.
  5. Loading→resolve flow. toast.loading() returns an id. Calling toast.resolve(id, { kind: 'success', title: '…', meta: '…' }) swaps the chip + body in place, applies the resolved kind's auto-dismiss window, and never remounts the toast (no flash).
  6. Auto-dismiss windows. success 4s · warn 7s · info 5s · undo 6s · error manual · loading until resolved or dismissed. Reduced-motion users get the same windows minus the progress animation.
  7. Stack cap. Up to 3 toasts visible. When a 4th arrives, the oldest dismissable toast (not error, not loading) gets trimmed FIFO. Errors and loading toasts are sticky.
  8. Position. Bottom-right on desktop. Bottom-inset-above-dock on mobile (uses safe-area-inset-bottom + dock height to clear the dock).
  9. A11y. success/info/loading/undo render with role="status" + aria-live="polite"; error/warn render with role="alert" + aria-live="assertive". Action and close buttons have aria-labels matching their visible text. Reduced-motion users get an opacity-only enter (no scale).
  10. Back-compat. Existing single-string call sites — toast.success('Logged in'), toast.error('Network error') — keep working unchanged. The string becomes the title; meta/action default to absent.
  11. Lint and tests pass.

Acceptance criteria

Logging in fires a success toast in the bottom-right with the new glass vessel + emerald chip. Auto-dismisses after 4s.

React concepts worth looking up

AnimatePresence with stable keys. The toast container uses Framer Motion's <AnimatePresence> to animate enter/exit when items appear or leave a list. Each toast needs a stable, unique key (the id we allocate) so React + Framer can match incoming and outgoing children. The trick for the resolve flow: keep the same key when a loading toast becomes a success — that way the component instance is preserved (no exit + enter animation, no DOM remount). Different key = remount + re-animate.
Functional state updates for queues. Pushing to / trimming a list of toasts from inside a callback means the latest state matters, not the closed-over state. Use setToasts(current => [...]) (the function form) rather than setToasts([...toasts, new]). Otherwise back-to-back fires will all read the same stale snapshot and overwrite each other. Equivalent of using ActiveRecord::Base.transaction + reload! instead of trusting a copy you grabbed before the write.
Refs vs state for id allocation. Each toast needs a unique id. useState would re-render every time you allocate one. useRef holds the counter without triggering renders. Same idea as a Rails class-level @@counter — keeps state across method calls without participating in the page's render cycle.
setTimeout cleanup in context providers. Auto-dismiss schedules a setTimeout. If the user manually dismisses or the toast gets trimmed first, the timer needs cancelling — otherwise it fires later and tries to remove a non-existent id (harmless, but wasteful, and can mask bugs). Track timers in a ref-keyed map and clearTimeout on every removal path, including unmount.
Tailwind v4 @utility blocks for non-utility CSS. The chip's spin animation, the gradient-text trick on the title em, and the warm-shadow recipe don't fit cleanly as Tailwind utilities. Add them as named @utility classes in globals.css (matching the hero-glow-urgent / marketing-halo-sunshine precedent) instead of inline style={{}} attributes. Inline-style is reserved for genuinely dynamic values (per-instance CSS custom properties like the auto-dismiss duration).
Polymorphic call signatures. The convenience methods take either a string or an options object: toast.success('Saved') and toast.success({ title: 'Saved', meta: '500ml logged' }) both work. Detect with typeof first === 'string' and normalise to the options shape. This is the JS analogue of Rails' flash[:notice] = 'string' vs flash.now[:notice] = '…' — same call site, slightly different payload, library handles both.

Notes & gotchas

  • Don't touch the notifications drawer. The drawer is TICKET-044 (R8). Mockup 30 + the "in context" stage of mockup 31 hint at it, but it has its own design surface, its own mark-as-read flow, and its own state lifecycle. This ticket only ships the floating toast.
  • Don't bake content sourcing into the toast. Some Plant Doctor / Tips messages may want richer markup later (links, inline icons). For now the title/meta/action shape is the contract — extending it is a future ticket's call.
  • Watch the back-compat surface. Every existing call site — including in tests — passes a string today. The polymorphic signature has to keep that working. If a test was depending on the old { message } shape internally, normalise it; don't rewrite the call sites.
  • Reduced-motion users still get the visual. No motion ≠ no toast. They get an instant fade-in/fade-out (no scale, no progress bar) but the same dismiss windows.
  • Mobile dock clearance. The dock is fixed at the bottom on mobile. Toasts position above it via bottom: calc(env(safe-area-inset-bottom) + dock-height + gap). If you change the dock height in TICKET-037 (R1 nav), this expression has to follow — track it via a CSS custom property if it starts drifting.
  • Closing checklist. When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on ticket-050-toast-redesign, then open a PR titled feat(v2): toast redesign (TICKET-050).
