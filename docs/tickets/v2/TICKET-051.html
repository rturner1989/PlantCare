Mental model: closest Rails analogy is "swap the ERB templates for redesigned ones, leave the controllers alone". Functional code = controllers + concerns; ERB = JSX + Tailwind. You're touching the ERB equivalent.

  5. Subtitle copy is warmer per mockup 29 — no generic "Enter your email below" form-app voice.
  6. Primary CTAs use the emerald-gradient lift (shadow-cta) consistent with the rest of the v2 app.
  7. Forgot/Reset success and error states are revisited — same content, post-21 visual quality.
  8. Functionality is unchanged: login + logout + register + forgot + reset all work exactly as they do today, including refresh-token cookie handling, RateLimitError → yellow toast, ValidationError → field-level errors, token-expired flow on reset, redirect-back via React Router state.
  9. Apple / Google sign-in is NOT added. The pages stay email + password only.
  10. All existing auth tests (Vitest + Playwright) stay green without modification — if a test fails, it's because a visual selector changed and the test needs a re-aim, not because behaviour broke.

  • Apple / Google sign-in. Benched per memory project_frontend_progress.md. Future post-launch ticket. Don't add provider buttons even as placeholders.
  • Functional changes. No new validation, no new rate-limit rules, no new error states, no behaviour tweaks. Diff is JSX + classes only (plus minimal copy edits).
  • Shared layout extraction unless duplication forces it. AuthLayout is 0 bytes today. Fill it only if R15's repaint reveals real shared chrome worth deduping. If you can repaint cleanly without it, leave the file empty (or delete it and follow up).
  • i18n / locale strings. Single-language English. See memory project_i18n_strategy.md — i18n deferred until real translation need surfaces.

React / Tailwind concepts worth looking up

Visual-only PRs still need to think about diff scope. The temptation in a "just visual" ticket is to also clean up adjacent things while you're in the file. Resist unless the cleanup is genuinely trivial and improves the same diff. Bigger refactors get spun out as follow-ups so the visual review stays scannable. Rule of thumb: if the cleanup needs its own commit message, it's a follow-up.
Tailwind 4 paper tokens. Post-F1, bg-paper, bg-paper-deep, bg-paper-edge all generate from --color-paper* in the @theme block. bg-card still works but resolves to paper via the alias. Prefer bg-paper in new auth markup — clearer intent — but don't sweep bg-card elsewhere in this PR.
Fraunces inline italic inside a sans heading. Mockup 29's headlines mix fonts via an inline <em> styled with font-display + italic. The rest of the heading stays sans. Pattern is the same one used on Step 1 of the welcome wizard — borrow that approach. Don't reach for a new typography primitive; one-off inline span is fine for the four auth screens.
Test selectors break when copy changes. Playwright auth specs target text like getByRole('button', { name: /Log in/i }). If you re-word the CTA, those selectors stop matching. Either keep the canonical button label ("Log in", "Create account") or update the test selectors in lock-step. Same applies to getByLabel('Email') if a label string drifts.
useFormSubmit is the canonical form-submit hook. Reminder, in case you reach for useState + manual handleSubmit instead. Existing pattern: const { submitting, handleSubmit, formRef } = useFormSubmit({ action, errorMessage }). Visual repaint shouldn't touch this — it's a behaviour change, out of R15 scope.

Notes & gotchas

  • The "no behaviour change" rule. If you find yourself touching AuthContext, useFormSubmit, ValidationError, RateLimitError, api/client.js, or any test that's not a selector re-aim — stop. That's R15 scope creep. Either revert and add a TODO, or split out as a separate ticket.
  • Refresh cookie + access token state. Don't tamper. Working today; visual ticket has no business near it.
  • Field labels stay canonical. "Email", "Password", "Confirm password", "Name". Tests + a11y both rely on stable accessible names. Reword display copy (heading + subtitle) freely; reword field labels only if you also re-aim every test selector that targets them.
  • Don't add a "remember me" toggle. The refresh-token cookie already provides persistent session. Adding a checkbox that does nothing or rewires the cookie behaviour is out of scope.
  • Don't add provider sign-in buttons. Apple / Google explicitly benched. Future post-launch ticket. No placeholders, no greyed-out "coming soon" buttons — they invite questions.
  • Reduced-motion. If you add an entrance animation to the auth card (mockup 29 doesn't show one but a subtle one would feel right), gate it on useReducedMotion() from motion/react. Existing pattern in the welcome wizard — same approach.
  • Mockup desktop = browser-frame demo. The "marketing column" is presentation only. App ships the form-card, centred on the gradient backdrop, with a tiny brand mark in the corner per the mockup's mobile tiny-brand pattern. Don't try to ship the marketing column.
