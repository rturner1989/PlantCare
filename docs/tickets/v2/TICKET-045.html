Onboarding intent-adaptive wizard — 8 steps, 4 intents

Open High

Summary

Rewrite the existing 5-step linear Welcome wizard into the v2 8-step intent-adaptive flow per mockup 19. User picks one of four intents (Forgetful · New parent · Sick plant · Browsing) at Step 1; per-intent config drives copy, optional step skips, default form values, and Step 7's CTA target. User.onboarding_intent + onboarding_step_reached persist on the server (B2, already merged) — refresh + re-login resume from where the user left off, indefinitely (no cleanup worker).

In one sentence: one wizard component, four intent paths, server-persisted progress so onboarding feels personal and forgiving.

Wizard state machine

{
  step: 0..7,
  intent: 'forgetful' | 'just_starting' | 'sick_plant' | 'browsing' | null,
  spaces: [{ id, name }],   // server-source-of-truth via useSpaces
  plants: [{ id, ... }],     // server-source-of-truth via usePlants
  // Form-input drafts that haven't yet committed (e.g. typing in the
  // species search box) are local useState in their step component —
  // not in the reducer. They evaporate on refresh; that's fine.
}

next()      → advance step, accounting for intent-config skips
back()      → retreat step, also respecting skips
goTo(n)     → URL-driven (refresh + back-button)
setIntent() → fires PATCH /profile, then next()

React concepts worth looking up

useReducer for wizard state machines. useState per field works fine for short forms. A wizard with branching, skips, and replayable transitions is a state machine — model it as one. useReducer takes (state, action) => nextState; you dispatch { type: 'next' } or { type: 'setIntent', payload: 'forgetful' }; the reducer encodes the rules. Equivalent to a Rails state machine gem (AASM): one place to read all transitions, no branching scattered across components.
Route state for "where did you come from". React Router lets you pass state through navigate(path, { state: ... }). The receiving page reads useLocation().state. Useful here for "log in mid-wizard → bounce back to your step" — Login already does this for the /welcome redirect path. Don't store wizard state in URL state; use it for "remember the intended destination" only.
TanStack Query as the wizard's "did this commit?" source. Step 2 commits each Space via useMutation. Step 3 reads usePlants() to show added plants. Step 4 walks through useSpaces(). Server is the source of truth — useReducer just tracks step + intent + UI flags. Mutations invalidate the relevant query keys; subsequent reads pick up the new state automatically. No manual sync.
StrictMode-safe step advancement. Every Next click that fires a PATCH /profile (to bump step_reached) needs to be idempotent — StrictMode double-invokes effects in dev, and the server should accept "already at step 5, you said go to 5" gracefully. Useful pattern: only advance if currentStep > previousStep; backwards advancement (back-button) doesn't write.
aria-live for step announcements. Sighted users see the step indicator update (1 of 8 → 2 of 8). Screen-reader users hear nothing unless we announce. Wrap the step heading in an aria-live="polite" region so each step transition reads aloud. Don't mark the entire wizard aria-live — too much chatter. Just the heading.

Notes & gotchas

  • No cleanup worker. Decided 2026-04-27. Resume from onboarding_step_reached indefinitely. Worker debate logged in conversation; if abandoned-onboarder rates become a real problem post-launch, revisit then with a soft-reset (clear step_reached, keep created records).
  • Server is source of truth for committed records. Don't reach into useReducer for the spaces list — use useSpaces(). Same for plants. The reducer tracks step + intent + transient UI flags only.
  • Every Next that bumps step_reached writes to the server. Backwards moves (Back button, browser back) don't write. The server's value is the high-water mark.
  • Sick plant intent skipping environment + stakes isn't conditional rendering scattered across step components — the orchestrator's next() consults intentConfig.skipSteps and jumps. Keeps the skip rules in one place.
  • Step 7 CTA fallback. Doctor route (R13) and Encyclopedia (R6) may not exist yet when R9 ships. If intentConfig.completionRoute resolves to a route that doesn't exist, fall back to /. Sick-plant + Browsing intents land on Today until their target routes ship.
  • Don't auto-write the welcome toast on completion here. R10 (TICKET-046) owns the welcome toast + driver.js tour kickoff via useFirstRunReveal. R9's job ends with POST /api/v1/onboarding/completion + navigate.
  • Apple/Google sign-in still benched. Doesn't apply here — wizard is post-auth.
  • Closing checklist (per PR): ./scripts/lint.sh && ./scripts/run_tests.sh client, branch-prefix ticket-045-onboarding-wizard-NN for sub-PRs, PR title feat(v2): R9 onboarding — <phase> (TICKET-045).
