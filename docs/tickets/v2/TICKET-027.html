React / Tailwind concepts worth looking up

Tailwind v4's @theme block. v4 dropped tailwind.config.js in favour of CSS-native theming. Custom properties under @theme { --color-* / --radius-* / --font-* / --shadow-* } are both CSS variables AND Tailwind utility generators. Define --color-paper: #fbf8f1 and you get bg-paper, text-paper, border-paper for free. The naming prefix is the contract — --color-* generates colour utilities, --radius-* generates rounded utilities, etc. CRA-era Tailwind users will find this surprisingly clean.
CSS variables are inherited at runtime — they're not compile-time constants. Changing --color-paper in DevTools live-updates every consumer. Different from Sass variables. Useful for testing visual regression: temporarily swap a token in DevTools to confirm a component reads from it (vs hardcoding the hex).
Two parallel declarations: @theme for Tailwind + :root for raw CSS. The current globals.css declares the same colours twice — once in @theme (so Tailwind generates utilities) and once in :root (so plain CSS, e.g. inside named @utility blocks like hero-glow-urgent, can read them). Keep this pattern. Adding a new token means two lines, not one.
Tailwind arbitrary values are an escape hatch, not a habit. rounded-[14px] works but bypasses the token. rounded-md reads from --radius-md. The audit step looks for arbitrary-value radii / blurs precisely because they're places the token system was ignored. Migrate when trivial; leave with a WHY comment when intentional.
Why CSS-variable design tokens beat JS theme objects. No re-render cost on theme change (it's CSS), no runtime JS dependency, dark-mode is a media-query :root override away, prefers-reduced-motion / forced-colors hooks for free. PlantCare doesn't use dark mode today but the architecture is in place if/when.

Notes & gotchas

  • Don't retire --color-card or --color-base blindly. Existing components read bg-card + bg-base in many places. Fastest non-regressive path: keep the variable names, point them at paper-equivalents (--color-card: var(--color-paper)), and let downstream tickets migrate consumers to bg-paper as their region work touches the file. Pure rename = wide diff = scope creep.
  • Tailwind rounded-md picks up its size from --radius-md. Changing the token from 12px to 14px updates every rounded-md consumer automatically. No diff in component files needed for the value bump itself — that's the point of tokens.
  • Bottom-sheet 28px / 24px lip stays hardcoded. Per spec §5, these are off-canon. Don't invent a --radius-sheet just for two consumers; if the value proliferates beyond Vaul drawers, F-tier follow-up can promote it.
  • --heat is reserved. Add it to the paper palette section as a commented-out placeholder with a one-liner explaining: "Phase 2 weather features will set this. No canonical hex yet — don't invent one." Avoids the "I see --sky + --frost but no heat, am I missing something?" question.
  • WCAG check the paper palette specifically. The previous palette was on white (#ffffff); paper (#fbf8f1) is darker and warmer. Coral on paper, sunshine on paper, ink-soft on paper — re-verify each combination passes 4.5:1. coral-deep on paper ≈ 4.7:1 (close to threshold, eyeball it). The /accessibility pass is the gate.
  • Don't add --radius-sheet or --blur-scrim speculatively. Tokens earn their slot when 2+ consumers need them. Spec §5 names what's canonical; everything else stays hardcoded until the second use case appears.
  • PR title: use the new convention — feat(v2): F1 design tokens (TICKET-027). See memory feedback_pr_title_v2_prefix.md.
