Onboarding intent + step-reached columns on User

Open High Backend-only

Summary

Persist User.onboarding_intent (string-backed enum: forgetful / just_starting / sick_plant / browsing) and User.onboarding_step_reached (integer, default 0) on the users table. Permit them on registration + profile updates. Expose both via User#as_json. Backend-only — no client work in this ticket.

The wizard UI that picks the intent (mockup 19, step 2) lands in TICKET-045 (R9). This ticket only ships the persistence layer so R9 has somewhere to write.

In one sentence: add two columns to users, an enum, and the four-line plumbing to persist + serialise them — nothing else.

Requirements

  1. Users have an onboarding_intent attribute that accepts one of four values (forgetful, just_starting, sick_plant, browsing) or nil.
  2. Users have an onboarding_step_reached attribute that defaults to 0 and accepts any non-negative integer.
  3. Both fields can be set on registration (POST /api/v1/registration) and updated on profile (PATCH /api/v1/profile).
  4. Both fields appear in the JSON returned by User#as_json (so the client can read them after login or refresh).
  5. Setting an invalid intent value (anything outside the four symbols) is rejected with a validation error, not silently coerced.
  6. Existing users — fixtures and any seed data — remain valid after the migration with intent = nil and step_reached = 0.
  7. The four intent symbol keys + their display labels live in a single USER_INTENT_LABELS constant on the model so future readers don't have to hunt for them.

Rails patterns worth getting right

This ticket has zero React work. The "concepts to get right" are all Rails-side:

String-backed enum convention. Project standard (CLAUDE.md): enum onboarding_intent: { forgetful: 'forgetful', ... } stores the symbol name as a string in a varchar column, not an integer. Pros: readable rows in psql, immune to symbol-key reorder breaking historical data, easy to compare from raw SQL. Cons: 8 bytes per row vs 1 byte. Rob's call long-since made — match it.
Label constant alongside the enum. The four symbols are domain-meaningful internally; the display strings ("I keep forgetting", "I'm just starting", etc.) are user-facing. Keep them in a single USER_INTENT_LABELS = { forgetful: 'I keep forgetting', ... }.freeze constant on the User model so callers — including the future R9 wizard's serializer — read from one source. Same pattern Rob's already using elsewhere (e.g. GAME_SYSTEM_LABELS per CLAUDE.md global standards).
params.expect, not permit. Rails 8 way per project convention. Both registrations + profiles already use params.expect(user: [...]) — just extend the symbol list. Don't introduce params.require(...).permit(...) alongside.
Counter-intuitive enum gotcha. Once an enum is declared, ActiveRecord auto-generates predicate methods (user.forgetful?) and bang setters (user.forgetful!). This can clash with existing methods or with future ones — the symbol key browsing is fine, but if a future enum uses keys that overlap an existing method on User (e.g. a save intent), save? would shadow ActiveRecord. Not a problem here, but worth knowing while you're naming.
Default migration shape. add_column :users, :onboarding_step_reached, :integer, default: 0, null: false. Using null: false + a default means existing rows get backfilled to 0 automatically. onboarding_intent is :string with no default, nullable — null is the meaningful "hasn't picked yet" state.

Notes & gotchas

  • Don't re-add onboarding_completed_at. Already exists from migration 20260416082612. Plan §B2 mentions it as a "merge with existing flag" reminder — meaning leave it alone.
  • String-backed enum syntax. Rails 7+ enum :name, { key: 'key' } form. The hash values are the strings stored in the DB. This is the project convention — don't fall back to integer-backed (enum :name, [:a, :b, :c]) which writes 0/1/2 to the column.
  • Skip prefix: if it conflicts. If browsing? ever clashes with another User method later, add prefix: true to namespace the predicate methods (onboarding_intent_browsing?). Today there's no conflict — your call.
  • Don't permit step_reached arbitrarily large or negative. Add a model-level validates :onboarding_step_reached, numericality: { greater_than_or_equal_to: 0, only_integer: true }. Stops a buggy client from writing -1 or 999.
  • Scope tests. Existing controller tests assert current_user isolation. The two new fields don't change that, but make sure new tests still go through auth_headers(@user) rather than skipping auth.
  • Don't ship Sidekiq jobs or notifications gating yet. Those branch on intent but live in their own region tickets (R8, R10). This ticket only adds the columns.
