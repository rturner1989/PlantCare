TICKET-039c · R3c

Unified page-aware search system

Open High

Summary

One search system shared across the app. A new SearchContext holds the current page's search scope (placeholder, query, setter, optional results renderer). Each page declares its scope on mount via useRegisterSearchScope; the sidebar (desktop) and mobile top-bar both read from the same context so a single keystroke flows wherever it's relevant.

The desktop sidebar's currently-disabled "Search…" pill becomes a real, focusable input. On mobile, the top-bar's disabled magnifier becomes a trigger that opens a full-screen search drawer per the v2 mockup. House is the first consumer — its existing ListView SearchBar is dropped in favour of the shared context, but the URL-driven ?space_id=N filter behaviour is unchanged.

In one sentence: page-aware search anywhere, owned by a context, with sidebar + mobile drawer chrome and House wired through as the first consumer.

What came before. R3a (TICKET-039a) shipped the House Rooms grid + chrome (PageHeader, SegmentedControl extensions, RoomCard, AddCustomSpaceForm). R3b (TICKET-039b) added the List view with a per-list SearchBar, Accordion per space, URL-driven view + filter via useSearchParams, and sessionStorage view persistence. The List view's local SearchBar was always understood to be temporary chrome until this ticket lifts search to the app shell.

What this unlocks. Any future page can plug into the same search surface by registering a scope on mount: Today (search tasks), Encyclopedia (search articles), Plant Detail (search care logs). Today + Encyclopedia don't get search added in this ticket — only the plumbing is built so they can wire in cheaply later. The mobile drawer becomes the search affordance for the entire app once it ships.

Mental model. Like before_action :set_breadcrumb in a Rails controller — pages declare what the chrome should show on their watch, the chrome reads it. The page is the source of truth for what's searchable; the chrome is dumb.

What you're building

Cross-cutting infrastructure (one context, one hook, one drawer overlay) plus the consumers that wire it up.

Contextclient/src/context/SearchContext.jsx
Holds the current scope: { placeholder, query, setQuery, isActive }. Pages register/unregister via the provider's registerScope(scope) method. When no page has registered, isActive is false and the chrome renders a disabled "Search… (coming soon-ish)" placeholder. Mounted under AuthProvider in App.jsx so authenticated pages share one source of truth.
Hookclient/src/hooks/useSearch.js
Reader hook for chrome. Returns { placeholder, query, setQuery, isActive, openMobileDrawer, closeMobileDrawer, isMobileDrawerOpen }. Sidebar input + mobile trigger + drawer all consume this.
Hookclient/src/hooks/useRegisterSearchScope.js
Writer hook for pages. Call once with the scope object on mount; effect cleanup unregisters on unmount. Accepts { placeholder, query, setQuery }. Stable identity expected — pass useState's setter directly so the effect doesn't churn.

Lifted from spaces/list/SearchBar.jsx verbatim — the pill-shaped input with magnifying-glass icon, sr-only label, conditional Clear-X (clears query + any registered onClearAll). Thin generic primitive; no domain knowledge.

Requirements

  1. When the user is on a page that has registered a search scope (House List view in this ticket), the sidebar's search pill is a focusable input whose placeholder + value reflect that scope. Typing updates the page's search results live.
  2. When the user is on a page that has NOT registered a scope (Today, Encyclopedia, Plant Detail), the sidebar's search pill shows a muted "Search… (coming soon)" state and is non-interactive.
  3. On mobile, tapping the top-bar magnifier on an active-scope page opens a full-screen drawer with the same input + the page's results visible underneath. The drawer closes via Escape, an X button, or tapping outside.
  4. Cmd+K (Ctrl+K on non-Mac) focuses the sidebar search input when a scope is active. The shortcut is a no-op when inactive.
  5. House List view's existing search behaviour is identical to before from the user's perspective — query filters plants live, clear-X clears query, scope-aware placeholder shows "Search Living Room…" when filtered to a space.
  6. Switching between pages cleans up correctly: leaving House (the only consumer) reverts the chrome to inactive state. No stale query bleeds across pages.
  7. The cross-cutting code is reviewed via /vercel-react-best-practices + /accessibility. /dhh-rails-reviewer is N/A — no backend changes.
  8. Lint and tests pass.

React concepts worth looking up

Modern context patterns + when context is enough vs when you need a store. For low-frequency state shared across a few siblings (sidebar, mobile bar, drawer, page), Context is fine. The "every consumer re-renders on every change" gotcha applies — but search query updates are user-driven keystrokes, not high-frequency animations, so it's not a real cost. If this grew into a global store with many independent slices (theme + auth + search + …), you'd reach for Zustand or similar — not yet.
Effect cleanup + unmount semantics. useRegisterSearchScope registers on mount and unregisters on cleanup. StrictMode runs effects twice in dev — your register/unregister has to be idempotent. Returning the unregister function from the effect is the standard pattern, same shape as adding/removing a DOM event listener.
Ref vs state for callback identity. setQuery passed into useRegisterSearchScope needs stable identity, otherwise the effect re-runs every render. useState's setter is already stable — pass it directly without wrapping. If you ever need to wrap, useCallback([]) or a ref are the tools, but neither is needed here.
Focus trap + the WAI-ARIA modal pattern. The mobile drawer is a modal. Tab/Shift+Tab must cycle within it, Escape must close it, focus must go to the input on open and back to the trigger on close. Same shape as Dialog.jsx already implements — copy that pattern, don't reinvent.
Keyboard shortcuts inside an SPA. A global Cmd/Ctrl+K listener attaches in a useEffect on mount of the provider (or a small hook below it). Watch out for conflicts: if the user is typing in a regular input and presses Cmd+K, you still want it to focus the sidebar. preventDefault on the keydown event blocks browser default (Chrome address bar focus on Cmd+K).

Notes & gotchas

  • One provider, one source of truth. Don't fork into desktop-only and mobile-only contexts — both consume the same SearchContext. The drawer is just chrome, not a separate state owner.
  • Effect cleanup. useRegisterSearchScope must unregister on unmount. StrictMode dev-mode double-invokes effects, so the unregister has to leave the registry in a clean state if called with a stale scope. Use a key (e.g. random id) or check identity before unregistering.
  • Cmd+K conflicts with the browser. Chrome focuses the address bar on Cmd+K by default. event.preventDefault() in the handler blocks it, but only when our handler runs first — attach to document in capture phase if you see drift.
  • Don't add scope to Today/Encyclopedia/Plant Detail in this ticket. Those are separate tickets so the search context can be reviewed in isolation. Scope creep here means a sprawling PR that's harder to review.
  • Don't wire backend search. R3c is client-side filtering of already-loaded data. Cross-page search and server-side search are future work.
  • Step 1 reorg risk. Lifting list/SearchBar.jsx to search/SearchInput.jsx means deleting the old file. Make sure ListView's import is updated and you didn't have any dead consumers (grep first).
  • Mobile drawer is a Dialog. Reuse the existing Dialog primitive's focus-trap + Escape pattern; don't roll a new one.
  • Three-tier review pass before merge. Run /vercel-react-best-practices + /accessibility on the cross-cutting code (context, hook, drawer, sidebar wiring). DHH n/a — no backend.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on branch ticket-039c-unified-search, then open a PR with title feat(v2): R3c unified page-aware search (TICKET-039c). No Claude footer in body or commit messages.
