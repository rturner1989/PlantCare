Edit / Log Care / Delete Plant dialogs

Open Priority: High

Summary

Plant Detail's overflow menu currently fires toast.info('… coming soon') for Edit, Log care, and Delete. R12 wires three dialogs into those entries, sharing the F10 dialog shell already established by R11 (Add Plant). Each dialog is a one-job CRUD surface against an existing plant — edit metadata, log a watering or feeding with date+notes, or delete the plant.

In one sentence: turn the Plant Detail overflow menu from cosmetic into functional by shipping three dialogs that mutate the plant on screen.

Requirements

  1. Edit plant from the overflow menu opens a dialog pre-filled with the plant's current nickname and space; saving persists the change and the Plant Detail page reflects it without a manual refresh.
  2. Log care from the overflow menu opens a dialog where the user picks Water or Feed, optionally adjusts the date (default = now), optionally writes notes, and submits to create a CareLog. The plant's care timestamps update on screen.
  3. Delete plant from the overflow menu opens a destructive confirm dialog naming the plant. For plants older than 30 days, Confirm stays disabled until the user types the plant nickname. Confirm hard-deletes the plant and lands the user on /house.
  4. Each dialog uses the F10 Dialog shell with Card.Header / Card.Body / Card.Footer as direct children — no double-Card wrapping. Title style matches the CRUD-dialog convention (plain bold paragraph, text-lg font-extrabold text-ink) — not Heading variant="display".
  5. Mutation hooks invalidate the relevant TanStack Query keys after success; no manual refetch from the consumer. Delete additionally navigates away.
  6. Visuals match mockups 34, 35, 36 — read CSS from the HTML files, not just the rendered iframe.
  7. Three review gates pass: /vercel-react-best-practices, /accessibility, /dhh-rails-reviewer.

Acceptance criteria

Open Plant Detail, tap overflow → Edit. Dialog shows current nickname + space pre-filled. Change nickname, Save. Toast confirms; dialog closes; the page heading updates without manual refresh.
Open overflow → Log care. Default care type is Water; date defaults to today. Switch to Feed, change date to yesterday, add a note, Save. Toast confirms; dialog closes. Care rings + last-care timestamps refresh on screen.
Open overflow → Delete on a plant created <30 days ago. Dialog shows "Delete {nickname}?" with Cancel + Delete. Confirm fires immediately (no type-to-confirm gate). Lands on /house; the plant is gone from the list.
Open Delete on a plant older than 30 days. Confirm button disabled. Type the wrong name → still disabled. Type the exact nickname → enabled. Confirm deletes; lands on /house.
Each dialog: Esc closes; clicking outside closes (matches existing Dialog primitive); focus-trap keeps Tab inside; first focusable element is Cancel on Delete (a11y rule for destructive actions).
DevTools → Network: submitting Edit fires PATCH /api/v1/plants/:id; Log care fires POST /api/v1/plants/:id/care_logs; Delete fires DELETE /api/v1/plants/:id. After Edit, the plant query refetches; ['plants'] list cache also refreshes.
Mobile (≤640px): all three dialogs fit the viewport, the form scrolls inside Card.Body if content exceeds height, the close-X stays reachable.
Lint and tests pass: ./scripts/lint.sh + ./scripts/run_tests.sh client.

React concepts worth looking up

TanStack Query mutations + cache invalidation. A mutation is a write (useMutation) — opposite of useQuery. After it succeeds, you invalidate the query keys whose data is now stale, and TanStack refetches them automatically the next time a component reads. Project rule (CLAUDE.md): invalidateQueries is the default; setQueriesData is a targeted optimisation only when you can name the specific race. For Edit and Delete, plain invalidate is the right call — both have cascading effects (plant list, dashboard, possibly Today). Rails analog: a controller that does plant.update; redirect_to plants_path, where the redirect is replaced by the cache invalidation.
Focus management for destructive dialogs. Accessibility rule the /accessibility review will flag: when a dialog confirms a destructive action, the initial focus must land on the safe choice (Cancel), not the destructive one. Reduces accidental Enter-key deletes. The Dialog primitive supports an initialFocusRef or similar — check components/ui/Dialog.jsx for the API. Also: Esc must close; the focus-trap in the Dialog primitive already handles Tab containment.
Controlled forms — error state shape. Multi-field error state is { field, message }, not a plain string (CLAUDE.md "Forms" section). The "type-to-confirm" gate on Delete is a controlled input — const [typed, setTyped] = useState(''), button disabled while typed !== plant.nickname. No need for the { field, message } shape on Delete — there's only one field — but Edit needs it (nickname, space).
Navigate-after-mutation pattern. On Delete success, the dialog should close AND the page should navigate away (the plant doesn't exist any more). Use useNavigate() from react-router and call it inside the mutation's onSuccess. Don't navigate before the request resolves — if the server rejects, you've lost the page mid-flight. Rails analog: plant.destroy; redirect_to root_path — except the redirect happens client-side after the response.
Three dialogs, one consumer — pick a state shape. Plant.jsx will need to know which dialog is open. Two clean options: three booleans (editOpen, logOpen, deleteOpen) — verbose but obvious; or one string (activeDialog: 'edit' | 'log' | 'delete' | null) — terser and makes "exactly one dialog open" structurally enforced. Either works. Pick by reading test — would a future reader prefer to grep for the boolean name or the discriminator? Don't reach for a reducer here.

Notes & gotchas

  • QuickDialog stays untouched. The one-tap RadialWheel care flow is a separate surface from LogCareDialog. Don't merge them; they have different jobs.
  • Doctor entry stays a placeholder. R13 owns it. Don't accidentally rip out the toast.info('Plant Doctor coming soon') handler on the doctor menu item.
  • EditPlantDialog locks species. Re-identifying a plant's species is a different flow (and a different ticket if it ever ships). Don't put a SpeciesPicker in Edit.
  • Personality quote stays visual-only. The confirm toast on LogCare can read a quote from personality/confirmQuotes.js, but it must not be framed as the plant speaking. Per memory project_voice_removed_from_plan.md and CLAUDE.md "Plant Personality" section: visual-only across all phases.
  • Type-to-confirm threshold is 30 days. Use the plant's created_at. Plants younger than that get a single-click Confirm.
  • Backend already supports all three. PATCH/DELETE /api/v1/plants/:id exist. POST /api/v1/plants/:id/care_logs exists with params.expect(care_log: [:care_type, :performed_at, :notes]). No backend work for R12.
  • CLAUDE.md cleanup rides this branch. Working tree already has M CLAUDE.md from the cleanup pass we did before scaffolding R12. Stage it as part of this ticket's first commit (separate commit on the same branch is fine — labelled chore(v2): CLAUDE.md cleanup pass).
