What you're building

component client/src/components/ui/SummarySlab.jsx
A two-piece compound primitive: <SummarySlab> wraps the column, <SummarySlab.Row> is each fixed-height row. Container is flex flex-col. Row carries min-h-[34px] lg:min-h-[62px] + a dashed top border on every row except the first. Decorative-only divider (CSS border, not <hr>, not in reading order).

Requirements

  1. SummarySlab renders a flex flex-col container that takes children unchanged.
  2. SummarySlab.Row locks min-height: 34px on mobile / 62px on desktop (lg breakpoint).
  3. The first Row has no top border. Every subsequent Row has a dashed border-paper-edge top border.
  4. Both pieces accept a className prop and pass through additional kwargs to the underlying div.
  5. SummarySlab.Row is reachable only via dot-access (SummarySlab.Row) — not exported as a named import. Keeps the API singular.
  6. The divider is decorative — no <hr>, no role="separator", no aria attributes. Pure CSS border.
  7. Lint and tests pass.

Acceptance criteria

Render a 2-row slab in a card; the Row heights are visibly identical and the dashed divider sits exactly between them.
Stack three SpaceCard-shaped slabs in a 3-column grid; the dashed dividers align horizontally across all three cards (the locked-height contract).
Resize the viewport across the lg breakpoint; row heights flip from 34px to 62px.
DevTools accessibility tree: no separator role, no hr element. The divider is invisible to screen readers.
Lint passes (./scripts/lint.sh).
Tests pass (./scripts/run_tests.sh client).

React concepts worth looking up

Static-property compound components. SummarySlab.Row is a function attached as a static property on the parent (SummarySlab.Row = Row). React idiom for child slots that always pair with a specific parent — see Tabs.Tab, Menu.Item. Cleaner than named exports because there's only one entry point. Compare to Rails ViewComponent's renders_many slot — same intent.
Tailwind v4 child-selector arbitrary classes. [&:not(:first-child)]:border-t emits CSS that targets every Row except the first. This is how we put the divider between rows without forking the component into FirstRow + Row variants. The browser handles which row gets the border based on actual sibling order — pure CSS, no JS.
Decorative-vs-semantic dividers. A CSS border-top is invisible to screen readers — it doesn't enter the reading order. An <hr> or role="separator" element does. Visual-only dividers should be borders; structural breaks (e.g. between major page sections) should be <hr>. SummarySlab's divider is purely visual, so border is correct.

Notes & gotchas

  • Don't expose Row as a named import. The static-property pattern is the singular API. Exporting both ways gives consumers two ways to do the same thing.
  • Don't add a Context. Row works standalone (no parent runtime check) — convention enforced, not technically. Same as Card.Header, Tabs.Tab. Adding a Context just to throw an error if Row is mounted outside the slab is overkill for a visual primitive.
  • Don't break the heights. 62px / 34px are locked per project_v2_design_decisions.md. They're not arbitrary — they're tuned so a card with two rows + a small header lands at the right total card height across mockups.
