Empty + Error states sweep

Open Priority: Medium

Summary

Closeout work, not new feature. Aligns 8+ existing empty-state surfaces against mockup 28 (per-region tone, hint slot, multi-action), introduces a full-page ErrorState primitive for 404/500, adds an app-level ErrorBoundary to catch render crashes, and a WidgetError for partial-region failures. Two memory entries close: project_empty_states_mockup.md + project_error_states_mockup.md.

In one sentence: stop apologising for the placeholder states — make every "nothing here" or "something broke" surface match the app's voice.

What came before. R11 (TICKET-047) shipped the Add Plant flow with its own one-off "no species" state. R12 (TICKET-049) wired Plant Detail's overflow menu and surfaced that CareView's "Recent care" panel still lacks an empty state for plants with zero logs. Across the app several surfaces still render bare strings or ad-hoc paragraphs for "nothing here" — Today, House, RoomsView, ListView (filtered), SpeciesPicker (search), NotificationsDrawer, Plant Detail. Mockup 28 catalogues the canonical empty-state shape; mockup 32 the error-state shapes. This ticket retrofits both everywhere they're missing.

What this unlocks. R13 Plant Doctor will need its own empty state ("no symptoms picked yet") — primitive will be ready. Future Achievements, Encyclopedia, and any new region inherits the extended EmptyState + new ErrorState. App-level ErrorBoundary means the next render crash blanks one widget instead of the whole page. The offline UX slice from mockup 32 (banner, cached-row labels, mutation queue, replay on reconnect) is deferred to its own ticket — service-worker + IndexedDB territory needs its own focus.

Mental model. Rails ships rescue_from ActiveRecord::RecordNotFound, with: :render_404 and render 'shared/error' partials. This ticket is the React equivalent for a SPA — <Route path="*" /> catches deleted records, ErrorBoundary rescues render crashes, EmptyState is the shared partial.

Requirements

  1. Every page-level "nothing here" state renders the extended EmptyState with the correct tone, an italicised emphasis word in the title, and the action set from mockup 28.
  2. Filter no-matches (ListView, SpeciesPicker, NotificationsDrawer filtered) show their distinct variant — different illustration, different actions ("Clear filters", "Browse popular", "Request species" etc.) — not a generic "nothing here".
  3. Visiting a deleted, non-existent, or typo'd URL renders the full-page ErrorState scheme="404" with Back-to-Today as the primary action — not a blank page, not the empty Layout chrome, not a generic React Router fallback.
  4. A render-time exception inside any routed page is caught by the app-level ErrorBoundary and replaced with ErrorState scheme="500" + Retry. The error doesn't blank the layout chrome (sidebar / topbar stay visible).
  5. Plant Detail's "Recent care" panel renders the "No care yet" empty state when the plant has zero CareLog entries. Logging one (via wheel water/feed or Log care dialog) clears the empty state and shows the log.
  6. The primitives are reusable — no copy-pasted inline error/empty markup left anywhere in pages/ or components/ after the sweep.

Add Plant dialog → search "zylarphia" (or any nonsense) → "We couldn't find "zylarphia"" with Browse-popular primary + Request-species secondary.

Visit /totally-fake-route → same full-page 404. Refreshing on the 404 page keeps you on 404 (not a blank screen).
Temporarily insert throw new Error('oops') at the top of a page component → 500 ErrorState renders inside the layout. Removing the throw + clicking Retry recovers the page.

Plant Detail on a freshly-added plant with no logs → CareView "Recent care" panel shows the empty state, not a blank row list.
Keyboard a11y: ErrorState's primary action receives initial focus on mount; WidgetError retry button is keyboard-reachable.

React concepts worth looking up

Error boundaries — still class-component territory. React 19 didn't change this: componentDidCatch + getDerivedStateFromError are the only way to catch render-time exceptions in descendants. Hooks can't replace this — there's no useErrorBoundary in core (TanStack Query has its own useQueryErrorResetBoundary for the query layer, but that's different). The class component is small; wrap it once at AppLayout and once per major widget that should fail isolated. Rails analog: rescue_from inside ApplicationController for app-wide + per-controller overrides for slice-level fallback.
Catch-all routes in React Router v7. <Route path="*" element=<ErrorState scheme="404" /> /> as the last child of <Routes> matches anything no earlier route matched. Lives inside the layout route so the 404 page keeps the persistent chrome (sidebar / topbar). Rails analog: match '*unmatched', to: 'errors#not_found' at the bottom of routes.rb — same pattern, same semantics, same "last route wins" rule.
Focus on mount for full-page error states. When 404/500 replaces the route content, keyboard focus is sitting on whatever the user last clicked — often a Link in the sidebar. Focus the primary action ("Back to Today") on mount so Enter recovers the user instantly. Same pattern as ConfirmDialog from TICKET-049 (requestAnimationFrame + ref). Don't use autoFocus — it doesn't survive React 19's concurrent mount + focuses too early.
ErrorBoundary placement matters. If the boundary wraps the whole router (above the layout), a render crash blanks the sidebar + topbar — defeating the point. Wrap <Outlet /> only so layout chrome survives. Per-widget boundaries go inside the page next to the widget they protect — Organiser widgets each get their own <ErrorBoundary fallback=<WidgetError /> /> so one fetch failure doesn't take down the dashboard.
ErrorBoundary reset across navigation. A boundary that has caught an error stays caught until something tells it to reset. React Router doesn't reset boundaries on route change by default. Pass key={location.pathname} on the boundary so it remounts (and re-renders fresh) when the route changes — or use TanStack's QueryErrorResetBoundary if the error came from a query. Otherwise the user is trapped on the 500 screen even after navigating elsewhere.

Notes & gotchas

  • Offline UX is a different ticket. No OfflineBanner / useOnlineStatus / cached-row labels / mutation queue here. Mockup 32's offline scene is real but it pairs with service-worker + IndexedDB queue work that warrants its own focus.
  • Don't add an external error reporter. No Honeybadger / Sentry / Bugsnag wiring inside componentDidCatch — that's an ops ticket. A console.error(error, info) in the catch is fine for now.
  • Achievements empty state is out of scope. Feature isn't built; the mockup's "streak starts on your first check-in" variant lands when Achievements does.
  • Mind the existing EmptyState callsites. Eight consumers today; the prop API change has to be backward-compatible OR migrate all eight in this ticket. Pick one approach and apply consistently.
  • ErrorBoundary inside AppLayout, not above it. If the boundary wraps the whole router, a render crash blanks the sidebar/topbar — defeating the point. Wrap <Outlet /> only so layout chrome survives.
  • Reset the boundary on route change. Pass key={location.pathname} so navigating away from a crashed page actually un-traps the user. Without this the user stays on the 500 screen forever even after clicking a sidebar link.
  • Test the 404 path on a real fake URL, not a mocked one. The catch-all route is easy to get wrong (route order, nested routes, layout wrapping). One Playwright spec hitting /totally-fake > a hundred unit tests.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, fire the three review skills (/vercel-react-best-practices, /accessibility, /dhh-rails-reviewer — last one is n/a here, no backend), commit on ticket-052-states-sweep, open a PR titled feat(v2): Empty + Error states sweep (TICKET-052).
