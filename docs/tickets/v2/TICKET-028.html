Requirements

  1. Track surface uses bg-paper-deep with 3px padding and gap, rounded-md.
  2. Active option renders on a bg-paper pill with shadow-warm-sm + rounded-xs; the pill animates between positions when selection changes (existing Framer layoutId pattern).
  3. Active option label is text-forest font-bold; inactive labels are text-ink-soft font-medium.
  4. Disabled options keep their opacity-50 cursor-not-allowed treatment and stay non-clickable.
  5. Existing keyboard navigation (left/right arrow walks through options via native radios) keeps working.
  6. Existing ARIA contract (role="radiogroup" + radio inputs labelled by their visible text) is preserved.
  7. Reduced-motion users get an instant pill swap, no spring animation.
  8. Lint and tests pass.

Design references

Mockups 17, 19, 21, 22, 35 all carry the canonical Segment recipe. The shape is identical across them — only the option labels change per region.

Spec §5 (design decisions): Segment = paper-deep track, paper-active-with-shadow-warm-sm. Used wherever a small set of options needs a single-pick toggle. FileTabs is its sibling for nested-folder UI; Segment is the default everywhere else.

React concepts worth looking up

Framer Motion layoutId for shared element transitions. The active-option pill uses a shared layoutId so the pill smoothly animates from the previously-active option to the newly-active one when selection changes. Each Segment instance generates its own layoutId via useId() — without that, two Segments on the same page (Step 4 stacks three) would all share one pill and animate weirdly between unrelated controls.
Native radios in a radiogroup beat custom buttons. The component looks like a button toggle, but each option is a hidden native <input type="radio"> with a styled <label>. That gives you keyboard arrow navigation, form submission semantics, and screen-reader announcements for free — no custom keyboard handler needed. Custom button impls would need aria-pressed + onKeyDown roll-your-own keyboard nav. This is the cheaper, more correct primitive.

Notes & gotchas

  • Don't change the API. Existing call sites (Step 4 onboarding, future House/Plant Detail) keep working unchanged. Props stay { label, value, onChange, options, icon, className }.
  • Don't migrate to button-based ARIA. The radio-based impl is more semantic and cheaper to maintain than aria-pressed button toggles. Plan mentions aria-pressed; the radio-group implementation is the better answer.
  • Pill rounded-sm → rounded-xs. The mockup pill is tighter than the track. --radius-xs (4px) lands the right inner-corner against the 14px track.
