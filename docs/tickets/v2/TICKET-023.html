TICKET-023 · v2 plan B4

Notifications system (Noticed gem foundation)

Open High Backend · Rails

Summary

Stand up the app's notifications foundation on top of the noticed gem. Phase 1 ships database delivery + ActionCable real-time + REST endpoint + two event sources (care due, milestone) + a daily Sidekiq cron job that drives them. Phase 2 (push notifications, email digest, AI doctor results, LLM plant messages) plug in as additional notifiers + deliver_by channels later without rewriting the surface area.

Originally specced as a hand-rolled Notification table; upgraded to Noticed after weighing Phase 2/2.5/3 plumbing cost, then ActionCable pulled into Phase 1 after weighing tech debt of adding it later. Then narrowed: self-action events (plant added, photo added, care logged) belong in the Journal region (R5/B3), not the notifications inbox — toasts already confirm those user-initiated actions immediately. The notifications inbox stays signal-rich for things the user might miss.

In one sentence: install Noticed, ship two notifier classes (CareDue + Milestone) with ActionCable delivery, expose GET /notifications + read/seen mutations, drive both via a daily Sidekiq cron job, and authenticate the cable channel via the existing refresh-token cookie.

  • Phase 2 web push — drops in as another deliver_by :web_push channel on the same notifiers. No new architecture.
  • Phase 2.5 LLM plant messages — ships through a future PlantMessageNotifier using the same envelope + drawer surface, broadcasting to the same channel.
  • Phase 3 weather alerts / sensor events — same pattern. Notifier-per-event-type scales.

Mental model: Noticed is to notifications what ActionMailer is to email. One framework-provided class per event type, framework owns the persistence + delivery dispatch. CLAUDE.md's "no service objects" rule still holds — Noticed::Event subclasses are framework-native abstractions, not hand-rolled service objects.

What you're building

Gem install, two tables (auto-migrated), four notifier classes, a daily cron job, REST controller, ActionCable connection + channel, User-model mixin, and tests.

ConfigGemfile
Add gem 'noticed', '~> 2.4'. Run bundle install + bin/rails noticed:install:migrations. Standard install path.
Migrationdb/migrate/*_create_noticed_tables.rb
Auto-generated by noticed:install:migrations. Creates noticed_events (event log: type, params jsonb, record_type, record_id, created_at) + noticed_notifications (per-recipient: event_id, recipient polymorphic, read_at, seen_at, created_at). Don't hand-edit beyond what the generator gives.
Modelapp/models/user.rb
Add has_many :notifications, as: :recipient, class_name: "Noticed::Notification", dependent: :destroy. Drop a def unread_notifications_count = notifications.unread.count helper that the controller can call without leaking Noticed::Notification into request specs.
Notifierapp/notifiers/care_due_notifier.rb
Fires when a plant's next_water_at or next_feed_at falls within a "due window" the daily cron job decides. deliver_by :database + deliver_by :action_cable, channel: "NotificationsChannel". Phase 2 chains deliver_by :web_push, if: -> { recipient.web_push_subscribed? }. Params: plant_id, plant_nickname, care_kind ('water' / 'feed'), days_overdue. Notification methods expose title, meta, url for the JSON serializer.
Notifierapp/notifiers/milestone_notifier.rb
Fires from the daily cron job when a plant hits 30 / 100 / 365 days since plant.created_at. deliver_by :database + :action_cable. Idempotent: cron job dedups by checking for an existing milestone notification with the same plant_id + day_count in the event log.
Channelapp/channels/application_cable/connection.rb
Authenticates WebSocket connections via the existing refresh-token httpOnly cookie. Reads cookie, verifies signature + expiry, identifies the connection by current_user. Rejects with reject_unauthorized_connection if the cookie is missing or invalid. Same auth surface as REST refresh, so no new token-management path.
Channelapp/channels/notifications_channel.rb
Per-user stream. def subscribed; stream_for current_user; end. Noticed's :action_cable deliverer broadcasts to this channel using the recipient as the stream key, so each user only receives their own notifications. Tiny class — most logic is in Noticed.
Jobapp/sidekiq/notifications_sweeper_job.rb
Runs daily (sidekiq-cron schedule). Walks Plant.includes(:user, :space), fires CareDueNotifier for any plant where care is due-today or overdue and no recent CareDue notification has fired (24h dedup window), fires MilestoneNotifier for plants matching anniversary windows. Adding sidekiq-cron is bundled if it's not already in the Gemfile.
Configconfig/sidekiq.yml (or schedule.yml)
Cron entry for NotificationsSweeperJob running daily — pick a quiet window (07:00 user-local rough proxy, settle for UTC for Phase 1; per-user timezone is a follow-up). Sidekiq-cron config block.
Controllerapp/controllers/api/v1/notifications_controller.rb
Inherits Api::V1::BaseController (auth applied). Three actions, all standard CRUD-shaped:
  • index — paginated list of current_user.notifications, newest first, plus unread_count. Default page size 20.
  • update — marks a single notification's read_at. Standard PATCH /notifications/:id.
  • seen — bulk-marks all unseen as seen. PATCH /notifications/seen via singular nested resource. Use this when the user opens the drawer.
All scoped through current_user.notifications.find_by(id:) per project security rule.
Configconfig/routes.rb
Adds resources :notifications, only: [:index, :update] + collection { patch :seen } inside :api, :v1 namespace.
Teststest/notifiers/*_notifier_test.rb
One per notifier (4 total). Asserts notification rows are created with the right params + recipient when triggered. Database delivery only — no channel mocking needed Phase 1.
Teststest/sidekiq/notifications_sweeper_job_test.rb
Asserts the job fires CareDue + Milestone notifiers for the right rows + skips already-notified plants within the dedup window. Time-travel via travel_to.
Teststest/controllers/api/v1/notifications_controller_test.rb
Index returns scoped to current_user, scope leak test (Bob can't see Alice's notifications), update marks read, seen bulk-marks. Standard request specs.
Teststest/channels/notifications_channel_test.rb
Channel auth: connection rejects without refresh cookie, accepts with valid one. Subscription streams scoped per user. Notifier broadcasts reach the right user's stream and not others (cable scope leak test).

Requirements

  1. The noticed gem is installed and its tables (noticed_events + noticed_notifications) exist in development, test, and production schemas.
  2. The daily sweeper job fires CareDueNotifier for plants where water or feed care is due within 24 hours OR overdue, scoped per user, with a 24-hour dedup window so the user doesn't get the same notification daily.
  3. The daily sweeper job fires MilestoneNotifier for plants whose created_at falls on a 30-day, 100-day, or 365-day anniversary today. Idempotent — re-running the same day produces zero duplicates.
  4. GET /api/v1/notifications returns the authenticated user's notifications, newest first, with unread_count separately. Page size 20 default. No leakage to other users (covered by scoped query).
  5. PATCH /api/v1/notifications/:id marks a notification as read for the current user. 404 if the notification belongs to another user.
  6. PATCH /api/v1/notifications/seen marks all the current user's unseen notifications as seen.
  7. Each notification serializes to a stable JSON envelope: id, type, title, meta, url, read_at, seen_at, created_at, params. Front-end can render every kind without case-statements per type.
  8. Each notifier delivers to :database + :action_cable in Phase 1, with a comment placeholder showing where Phase 2 channels (:web_push, :email) will plug in.
  9. The cable connection authenticates via the existing refresh-token httpOnly cookie. Connections without a valid cookie are rejected.
  10. Notifications channel scopes per recipient — user A's connection only receives broadcasts from notifiers where recipient == user_A.

Acceptance criteria

bin/rails db:migrate on a fresh checkout adds noticed_events + noticed_notifications with no errors.
Setting plant.next_water_at to today and running the sweeper job in rails console creates a CareDue notification for that plant. Running the sweeper a second time within 24 hours creates zero additional rows.
Travelling time forward 30 days and running the sweeper job creates a milestone notification for that plant. Running the sweeper a second time the same day creates zero additional rows.
A second user (B) calling GET /api/v1/notifications sees zero rows from user A — verified in request specs.
Calling PATCH /api/v1/notifications/seen while having 5 unseen notifications returns { unread_count: 5 } (still unread — seen ≠ read), and a follow-up GET reflects all seen_at timestamps populated.
Calling PATCH /api/v1/notifications/:id on one row drops unread_count by 1 on the next index call.
Sidekiq dashboard shows the daily cron job scheduled. Triggering it manually from the Sidekiq UI executes successfully.
Opening a WebSocket connection to /cable with a valid refresh-token cookie identifies the user. Without the cookie, the connection is rejected.
Triggering a notifier in rails console for user A causes a broadcast on user A's NotificationsChannel stream. User B subscribed to their own stream sees no message.
API tests green: notifier tests, sweeper-job test, controller tests including scope leak. Run via ./scripts/run_tests.sh api.

Notes & gotchas

  • Web Push stays out of scope. Web Push needs subscription management + VAPID keys + browser permission flow — own Phase 2 ticket. Just leave the deliver_by :web_push comment placeholder on each notifier.
  • ActionCable IS in scope here (ticket-scope upgraded — see Summary). Cable auth via refresh-token cookie; broadcasts reach NotificationsChannel per user. Frontend subscription wiring lives in the follow-up client PR.
  • Cable adapter check. Production needs Redis adapter (project already has Redis with db 1 reserved for ActionCable per existing config). Development can use the async adapter (default). Confirm config/cable.yml production block points at REDIS_CABLE_URL with a sensible default like the project's other Redis-uses pattern.
  • Sidekiq-cron is the recommended cron scheduler. Project already runs Sidekiq + Redis. Add gem 'sidekiq-cron' if not yet present and configure via config/schedule.yml loaded in a Sidekiq initializer.
  • The dedup window for CareDue. Use Noticed::Event.where(type: "CareDueNotifier", record: plant).where("created_at > ?", 24.hours.ago).exists? to skip re-firing. Don't write a new dedup table.
  • Don't dispatch notifiers from inside after_create. Use after_create_commit so the notifier fires after the DB transaction succeeds. after_create fires inside the transaction — if the outer save fails, the notification persists with no plant attached.
  • Time zones for the daily sweeper. Phase 1: run in UTC, accept that "due today" is approximate for users in distant time zones. Per-user TZ is a follow-up — note in code that the cron schedule is timezone-naive for Phase 1.
  • Polymorphic recipient. Don't restrict noticed_notifications.recipient to User in the migration. Leave polymorphic — Phase 2.5 might add per-Plant recipient (LLM messages voiced by a specific plant where the plant is the addressee for routing).
  • JSON shape: serializer or as_json? Per project convention (as_json on models, no serializer gems), put the JSON shape on the Notification class via a thin model concern or override as_json on a wrapper. Noticed::Notification ships its own as_json — extend rather than replace.
  • Rate-limit the controller? Not for this ticket. Notifications endpoint is read-heavy + cheap; if it gets abused, add Rack::Attack rule in a follow-up.
  • i18n. Hardcode notification titles in English on the notifier classes for Phase 1. Per project_i18n_strategy.md i18n is post-launch.
  • Frontend follow-up after this ticket merges (separate short PR): swap the disabled bell button stubs in Sidebar.jsx + MobileTopBar.jsx for live unread-count badges. Add @rails/actioncable npm dep, build a useNotifications hook (initial REST fetch via TanStack Query for backfill, ActionCable subscription invalidates the query on each new notification — no polling). Single-flight subscription owned by an app-level NotificationsProvider mounted under AuthProvider so the consumer is created once per session.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh api, run /dhh-rails-reviewer, then /pre-commit + /comment-audit, commit on this branch (ticket-023-notifications-endpoint), and open the PR with the feat(v2): B4 notifications system prefix.
