TICKET-044 · v2 plan R8

Notifications drawer + bell badge wiring

Open High Frontend · React + ActionCable

Summary

Wire the bell buttons (Sidebar + MobileTopBar) to a real notifications drawer. B4 (TICKET-023) just shipped the backend — Noticed gem, REST endpoints, ActionCable channel, daily sweeper. The bell stubs from R1 (TICKET-037) currently render disabled. This ticket flips them to live unread-count badges that open a drawer showing the current user's notifications.

Drawer is desktop-right-slide (420px, no scrim — Mac-notification-centre style) and mobile-right-slide (full-width). Each item shows kind-icon + title + meta + time-ago. Click marks-read and navigates to notification.url. Drawer-open fires a single mark-all-seen so the badge clears on glance even when items go un-clicked.

ActionCable subscription is owned by a single app-level NotificationsProvider, mounted under AuthProvider. Server broadcasts (sweeper firing, real-time events) push updates into the same TanStack Query cache that the bell + drawer read from — no polling.

In one sentence: swap the disabled bell stubs for a live drawer connected to the cable channel, with mark-read + mark-seen flows wired to B4's REST endpoints.

What came before: R1 (TICKET-037) shipped Sidebar + MobileTopBar with bell button stubs — disabled Action elements with aria-label="Notifications (coming soon)". B4 (TICKET-023) shipped the backend last week — Noticed gem, REST controller (GET /notifications, PATCH /notifications/:id, POST /notifications_seen), ActionCable channel scoped per-user via the existing refresh-token cookie, daily sweeper firing CareDue + Milestone notifiers. The system has data + a delivery channel; the UI consumes nothing.

What this unlocks: R2 Today (TICKET-038) has an organiser column on desktop that previews recent notifications — same hook + same context, no new fetch. The drawer pattern itself becomes a reference for any future right-slide panel work. Wiring the cable subscription proves the WS auth path end-to-end so Phase 2 work (Web Push, AI doctor messages) can land additional notifier classes without rewiring the consumer.

Mental model: Rails analogy is the layout's persistent flash partial — but instead of flash[:notice] rendering inline once, this is a persistent inbox the user can pull open at any time. The "drawer" is a piece of chrome that lives in the layout, owned by an outer provider, fed by a single query cache shared between the bell badge and the drawer body.

What you're building

Utilclient/src/lib/actionCable.js
Module-level singleton wrapping createConsumer('/api/v1/cable'). Lazy-initialised on first call so test environments that don't open WebSockets stay quiet. Exports a cableConsumer() getter.
Contextclient/src/context/NotificationsContext.jsx
App-level provider mounted under AuthProvider in App.jsx. Owns the single ActionCable subscription per session (creates the consumer + subscription on mount, tears down on unmount). Owns the drawer-open boolean state. Exposes { open, openDrawer, closeDrawer } via context. Hookclient/src/hooks/useNotifications.js
useQuery on ['notifications'] hitting GET /api/v1/notifications. staleTime: 30_000. Returns { data, isLoading, isError }. Anyone who needs unread count or list reads from here — Sidebar bell, MobileTopBar bell, drawer body. One fetch, multiple consumers, courtesy of TanStack Query's deduping.
Hookclient/src/hooks/useMarkNotificationRead.js
useMutation wrapping PATCH /api/v1/notifications/:id. On success: queryClient.invalidateQueries(['notifications']). Returns the mutate function. Called by NotificationItem on click.
Hookclient/src/hooks/useNotificationsSeen.js
useMutation wrapping POST /api/v1/notifications_seen. On success: invalidate ['notifications']. Fires from the drawer's open-effect — once per open, not on every render. Important: seen ≠ read. Seen says "you've glanced at it" (clears the badge subtle indicator); read says "you've acted on it" (per-item click).

Requirements

  1. The bell button in Sidebar (both Full and Drawer states) and MobileTopBar shows the live unread notification count when > 0.
  2. Clicking any bell opens the notifications drawer.
  3. The drawer renders the current user's notifications from GET /api/v1/notifications, newest first, flat list (no grouping).
  4. Each notification row shows kind icon, title, meta line, and a relative time-ago string. Unread rows are visually distinct from read rows.
  5. Clicking a notification row marks it read and navigates the user to the notification's url.
  6. The drawer has a "Mark all as read" button visible when there's at least one unread notification. Clicking it marks every visible notification as read.
  7. Opening the drawer fires POST /api/v1/notifications_seen exactly once per open (not per render). The mutation invalidates the cache so the next read reflects updated seen_at stamps.
  8. The drawer closes on ✕ button, Escape key, and (desktop only, since there's no scrim) clicking outside the drawer panel.
  9. An empty state renders when the user has zero notifications — calm copy, illustrative emoji, no list shell.
  10. An ActionCable subscription is established on app mount (after the user is authenticated) and torn down on unmount. Server broadcasts to the user's NotificationsChannel stream invalidate the ['notifications'] query so the bell and drawer update without polling.
  11. The drawer's open state survives navigation between protected routes (closing the drawer is an explicit user action, not a side effect of route change).

React concepts worth looking up

TanStack Query + WebSocket = invalidation glue, not state replacement. The pattern: REST useQuery is the source of truth. WebSocket broadcasts call queryClient.invalidateQueries on receive — that triggers a refetch which updates every consumer. You don't store the WebSocket payload in state. The cable is a "hey, refresh" signal, not a data delivery channel. Rails analogy: it's like Turbo Streams telling the browser to re-fetch a frame, not delivering the frame contents inline.
One subscription per session, not per component. If you put useEffect with consumer.subscriptions.create inside a hook that components call individually, you'll open N WebSockets — one per mounted consumer. The fix is a single app-level provider that owns the subscription. Components read shared state from the provider's context and shared data from the TanStack Query cache. Both are global by design; the WebSocket is the resource.
Context for state that crosses components, NOT for cached server data. The drawer-open boolean belongs in NotificationsContext because Sidebar, MobileTopBar, and the drawer itself all need to read and toggle it. The notifications array does NOT belong in context — it's server data, lives in TanStack Query's cache, and re-fetches via invalidation. Putting server data in context means re-implementing caching, deduping, and revalidation by hand.
Effect-once-per-open vs effect-on-render. Drawer-open should fire mark-seen once when it opens. Naive useEffect(() => markSeen(), [open]) fires on every render where open is referenced. Use a deps array of [open] (re-fires only when open changes) and gate inside with if (!open) return. Cleanup function isn't needed since the mutation fires once and resolves; if it were a subscription it would.
StrictMode double-fire of effects in dev. The mark-seen-on-open effect will fire twice in development StrictMode (mount → unmount → re-mount). Two POSTs to /notifications_seen are idempotent on the server (re-marking already-seen is a no-op), so this is safe. But verify: if the request count actually mattered (e.g. analytics events), you'd need a ref guard.
Click-outside-to-close — only on desktop, not on mobile. Desktop has no scrim, so clicking the main content area is the implicit "I'm done with the drawer" signal. Mobile is a full-width sheet — there is no main content area to click. Use a media-query check (or just gate by viewport size) inside the click-outside listener so it doesn't accidentally fire on mobile.

Notes & gotchas

  • Don't add a server-side bulk-read endpoint in this ticket. Phase 1 "Mark all as read" iterates client-side via Promise.all over useMarkNotificationRead.mutateAsync. With max ~20 items per page that's fine. If real users hit it hard, add PATCH /api/v1/notifications/read_all as a follow-up — it's a 6-line controller action.
  • Mark-seen vs mark-read confusion. Seen = "user opened the drawer" (clears the badge subtle indicator). Read = "user clicked the item" (clears the per-item dot). They're separate timestamps on the Noticed::Notification record. unread_count on the index endpoint returns unread.count — items with read_at IS NULL, regardless of seen state.
  • Cable consumer auth path is the existing refresh-token cookie. Don't pass tokens in the URL or via subprotocol. Browser auto-sends the cookie because cable mounts at /api/v1/cable (B4 path-aligned the cookie scope). If the consumer fails to connect, double-check the user is logged in and document.cookie shows refresh_token.
  • WSS in production rides the existing TLS termination. Don't hardcode wss:// or ws:// in the consumer. createConsumer('/api/v1/cable') with a relative path lets ActionCable's JS resolve the protocol from window.location.protocol. Same TLS cert as the rest of the API.
  • Drawer focus trap. When the drawer opens, focus should move to the close button. Tab cycles inside; Escape closes; close button restores focus to the bell that opened it. Reuse the focus-management pattern from Sidebar.jsx's drawer state — same shape.
  • Disabled bell tests in Sidebar.test.jsx + MobileTopBar.test.jsx need updating. They currently assert the bell is disabled with aria-label="Notifications (coming soon)". After this ticket those assertions flip — bell is enabled, label is "Notifications", clicking opens the drawer. Update assertions in the same commit so tests stay accurate.
  • Don't render the drawer inside the auth pages. Mount it inside AppLayout, not App.jsx's top-level. Auth pages have no bell to open it from + no notifications to show.
  • The provider mount order matters. NotificationsProvider goes under AuthProvider (so it can read the user) but above BrowserRouter (so the same provider serves every route). It does NOT belong inside AppLayout — that would re-mount the consumer on logout/login cycles.
  • Postman collection update for the 3 endpoints from B4 was deferred in TICKET-023's PR. Optional: bundle the Postman update into this ticket since you're touching the same endpoints. Or keep deferred.
