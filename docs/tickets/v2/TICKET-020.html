React concepts worth looking up

TanStack Query keys are identity, not just naming. When you change ['rooms'] to ['spaces'], you're not just renaming — you're invalidating the entire client-side cache for that resource on next mount. Any code that called queryClient.invalidateQueries(['rooms']) won't trigger refetches anymore. Grep for 'rooms' as a string literal across the client too — it's the kind of thing tests will catch only if they actually exercise the invalidation path.
Cache-key shape (CLAUDE.md, "Cache keys" section). Client uses flat arrays — one segment per array entry. ['spaces'], ['spaces', id]. Never nest (['spaces', ['detail', id]] = wrong). Never join into a string ('spaces:detail' = server-side syntax, not client). The shape is identical to the resource's Rails model name (singular noun).
Vite import paths are case-sensitive on Linux/CI. If you rename RoomCard.jsx → SpaceCard.jsx via the filesystem and any import keeps the old casing on a case-insensitive macOS APFS volume, local dev runs fine and CI explodes. Use git mv so git tracks the rename, and verify all importers update in the same commit.
The "phantom import" trap with renamed hooks. If you delete useRooms.js but leave any import { useRooms } from '../hooks/useRooms' elsewhere, Vite's build error will name the importer file — useful. But Vitest won't fail until the test loads that path. Run the full vitest suite (not just the renamed hook's test) before committing.
Rails analogy — fixtures. A Rails fixture file's name has to match the table name (rooms.yml → rooms table). After the rename, the fixture file must be spaces.yml; Rails auto-loads spaces(:living_room) based on the filename, not the YAML key. The internal YAML keys (living_room: …) are just identifiers for cross-fixture references — they don't have to change unless you want them to.

Notes & gotchas

  • Don't dual-emit JSON. No { room: ..., space: ... } "for compat". Pre-launch single-shot — the client and server change in lockstep.
  • Counter cache. plants_count on the rooms table moves with the table — the column name is plants_count, not rooms_plants_count, so no rename needed there. But verify schema.rb after migrate.
  • Don't touch db/migrate historical migrations. Old create_rooms + add_plants_count_to_rooms + add_unique_index_to_rooms_name_user_id stay as-is. The new migration on top brings the schema to its final state.
  • Welcome wizard step rename ripples. Step2Rooms is referenced from Welcome.jsx and probably the route table for the wizard. Rename the file, the import, the component identifier, and any test selectors that target data-testid="step2-rooms" or text like "Add a room".
  • Icon presets that include "bedroom" / "bathroom" stay. Those are values inside roomIcons, not the module name. After the rename to spaceIcons, the keys can stay — they're still valid space presets.
  • Playwright tests assert visible text. Anywhere a .spec.js does page.getByText('Rooms'), update it. Same for any aria-label assertions.
  • Don't bother adding a 410-Gone redirect from /api/v1/rooms. Pre-launch, no consumers — let it 404.
  • Counter cache sanity-check after migrate. Run Plant.connection.execute("SELECT space_id, COUNT(*) FROM plants GROUP BY space_id") and compare to spaces.plants_count. Matching = migration good. (Not a long-term concern; just paranoia for the rename PR.)
