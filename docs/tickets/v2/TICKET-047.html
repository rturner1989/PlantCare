Add Plant dialog

Open High

Summary

Replaces the placeholder /add-plant route with a real cross-page dialog that creates a plant scoped to a space. Two internal steps — species pick, then space + nickname + live schedule preview — wrapped in the Dialog primitive. Environment is NOT asked per plant; it lives on the space (R9 baseline) and the dialog reads it through. The Dock FAB, sidebar/topbar triggers, and per-space CTAs on House all open the same dialog via context, no full-page navigation.

In one sentence: Add Plant becomes a universal action available from anywhere, not a destination.

What came before. R3c (TICKET-039c, merged) shipped the unified search system + space CRUD (edit pencil + delete trash + ConfirmDialog primitive). House now has full space CRUD but no per-space "Add a plant" affordance — flagged in memory project_add_plant_flow_gap.md. Onboarding Step 3 (TICKET-045) already includes a working species-pick + create-plant flow against useCreatePlant; this ticket lifts that flow into a reusable dialog.

What this unlocks. R12 (TICKET-048: Edit / Log Care / Delete Plant dialogs) reuses the same Dialog + form primitives. R4 (TICKET-040: Plant Detail v2) becomes meaningful only once users can actually create plants — every /plants/:id link in the app currently dead-ends on a placeholder. The search drawer's empty state ("no plants match X") gets a future "Add plant called X" CTA once this dialog exists.

Mental model. This ticket is Rails' "open-modal-from-anywhere" pattern: a single AddPlantDialog registered at app root, its open/close state lives in a context (think flash[:open_modal]), and any nav link or button can call useAddPlant().open({ defaultSpaceId }) to invoke it — no route navigation, no prop drilling.

One context, one dialog component (composed of two internal step components), one app-level shortcut, plus the consumer wiring on House.

Requirements

  1. Add Plant is reachable from anywhere in the app — clicking the Dock FAB opens the dialog over the current page, no route navigation.
  2. The dialog walks the user through species pick → details (space + nickname + schedule preview) in two internal steps. Back/Continue buttons in the footer move between steps; submitting on step 2 creates the plant.
  3. When opened with a defaultSpaceId (e.g. House per-space "Add a plant" CTA), the space picker is pre-selected on step 2 and the schedule badges render immediately based on that space's environment.
  4. Step 2 shows live schedule badges — "Water every 7 days", "Feed every 30 days" — calculated from the picked species' base cadence × the selected space's env modifiers. Changing the space updates the badges before save.
  5. Saving fires a toast ("Added <nickname>"), closes the dialog, and lands the user on /plants/:id for the freshly-created plant.
  6. House Rooms grid + List view both expose a per-space "Add a plant" affordance. Clicking it opens the dialog with that space pre-filled.
  7. The placeholder /add-plant route is gone — visiting it directly redirects to the home page (or 404s gracefully).
  8. Pressing N with no input focused opens the dialog. Typing N inside any input does NOT trigger the shortcut.
  9. Closing the dialog mid-flow (Escape, X, backdrop click) discards in-progress state. Reopening starts fresh on step 1.
  10. Keyboard navigation works inside the dialog: Tab cycles within the dialog (focus trap), Escape closes, the input on each step gets focus on enter.
  11. Cross-cutting code is reviewed via /vercel-react-best-practices + /accessibility. /dhh-rails-reviewer is N/A — no backend changes.
  12. Lint and tests pass.

Acceptance criteria

Dock FAB click opens the dialog. Pressing Escape closes it. The URL never changes during open or close.
On step 1, typing "rose" in the species search returns matching results from Perenual within ~300ms (debounced). Picking a species advances to step 2.
On step 2 with no defaultSpaceId, no space is pre-selected. With defaultSpaceId, that space is highlighted on render and schedule badges show immediately.
Picking different spaces on step 2 updates the schedule badges live (e.g. "Bright" space shows shorter water cadence than "Dim" for a thirsty species).
Nickname is required — submitting empty shows an inline error on the field. Filling it + clicking Save creates the plant + closes the dialog + lands on /plants/<id>.
House Rooms grid: each room card has a "+" icon button next to edit/delete. Clicking it opens Add Plant with that space pre-selected on step 2.
House List view: each accordion summary has the same "+" cluster. Empty accordion body's "Add one" CTA opens the dialog (replaces the bare /add-plant link) with the space pre-filled.
Visiting /add-plant directly redirects to / (or hits the 404 page gracefully). The route is no longer registered.
Pressing N on Today opens the dialog. Pressing N inside the search input on House does NOT.
DevTools React tree: only one AddPlantProvider mounted. Multiple consumers (Dock, Sidebar, House CTAs, drawer) all read the same context.
Closing mid-flow (e.g. on step 2 with a species picked) → reopening shows step 1 with no species selected.
Lint and tests pass (./scripts/lint.sh && ./scripts/run_tests.sh client).

React concepts worth looking up

Modern context patterns + when to split. AddPlantContext follows the same shape as SearchContext: split into actions (open/close — stable refs) and state (isOpen, defaultSpaceId — changes on open) IF you find consumers re-rendering for no reason. For Add Plant, almost every consumer reads open only — they never need isOpen. So a single context is fine here; split if review flags fan-out.
Form state ownership across multi-step dialogs. Two reasonable approaches: (a) one useState object in AddPlantDialog holding all step values, drilled down to step components as props; (b) useReducer with action types per step (SET_SPECIES, SET_SPACE, SET_NICKNAME). For three steps with simple shape, (a) is fine. Reach for (b) only if validation cross-references multiple steps.
TanStack Query useMutation + cache invalidation. useCreatePlant already exists. Calling .mutate(data) fires the POST; onSuccess in the hook invalidates ['plants'], ['spaces'], ['dashboard'] so the new plant appears in House + Today without manual refetches. You don't have to do anything — TanStack handles the cascade.
Keyboard shortcuts inside an SPA. Document-level keydown listener attached in the provider's useEffect. Skip the handler when document.activeElement is an input / textarea / contenteditable, otherwise typing the letter into a search box would also open the dialog. Pattern mirrors the Cmd/Ctrl+K hoist in SearchContext.
Dialog primitive's mobile bottom-sheet behaviour. The existing Dialog auto-detects mobile viewport and switches to a bottom-sheet animation (drag-to-dismiss). Don't reinvent. Just pass the dialog's children — Dialog handles placement, focus trap, and focus return. The R3c MobileSearchDrawer reaffirmed the trap-and-return pattern.
StrictMode double-fire on provider mount. The provider's keydown listener attaches in dev twice in a row (StrictMode). Returning the cleanup from useEffect handles it — listener is removed and re-added between mount and the second pass, so you end up with one listener. Just make sure the cleanup runs.

Notes & gotchas

  • Don't duplicate the species-pick logic from onboarding Step 3. If the search + popular-species code is already extracted to a reusable component, import it. If not, extract during this ticket so onboarding + dialog share one source of truth.
  • Photo upload is OUT of scope. R12 (Edit Plant) handles it. Don't add a step for it here, even if mockup 27 shows one — defer to keep this ticket reviewable.
  • Don't build Plant Detail v2 here. Saving navigates to /plants/:id which is still the placeholder until R4. The link works; the destination is bare. That's fine — R4 lands soon.
  • Discard mid-flow state on close. No "draft" persistence. Closing the dialog wipes the form. Reopening = clean state. Mirrors the existing AddCustomSpaceForm + Dialog primitives' behaviour.
  • Watch for nested Dialog when "Add new space" is invoked from step 2. AddCustomSpaceForm uses Dialog. Opening it on top of AddPlantDialog stacks two Dialogs. Either: (a) close AddPlantDialog before opening AddCustomSpaceForm and reopen on completion, or (b) accept stacking and verify focus trap behaves. Test both paths before merge.
  • Three-tier review pass before merge. Run /vercel-react-best-practices + /accessibility on the new context, hook, dialog, and consumer wiring. DHH n/a.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on branch ticket-047-add-plant-dialog, then open a PR with title feat(v2): R11 Add Plant dialog (TICKET-047). No Claude footer in body or commit messages.
