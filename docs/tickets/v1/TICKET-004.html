TICKET-004

App shell — router, protected routes, responsive nav

Open High

Summary

Build the persistent frame — the sidebar, dock, and router — that every page in PlantCare will live inside. A user refreshing the app either lands in the logged-in shell (sidebar on desktop, glass dock on mobile) or gets bounced to /login. Clicking any nav item routes to a placeholder page for now — future tickets replace those placeholders with real screens one at a time.

In one sentence: a user can log in, see an empty-but-navigable app, and get a proper 404 if they wander off the map.

What you're building

Seven concrete artifacts. When this ticket is done, each of these will exist and have a clear job:

rewriteclient/src/App.jsx
The top-level component that wires everything together: QueryClientProvider → AuthProvider → BrowserRouter → <Routes>. Currently just renders <div>PlantCare</div>; becomes the router declaration and provider tree. Nothing else renders pages — this file is the map.
componentclient/src/components/ProtectedRoute.jsx
A small wrapper component whose only job is "am I logged in?". If yes, it renders its children. If no, it redirects to /login — and remembers where the user was trying to go so Login can bounce them back after they authenticate. While the initial session-restore is still in flight, it shows a spinner instead of either state (this is what prevents the login-screen flash on refresh).
componentclient/src/components/Layout.jsx
The persistent "frame" that wraps every authenticated page. Renders the Sidebar, the Dock, and a <main> area containing React Router's <Outlet /> — which is the slot where the current page's content appears. When you navigate from Today → House, only the Outlet's contents change; the sidebar and dock stay put.
componentclient/src/components/Dock.jsx
The mobile-only floating glass nav bar, pinned to the bottom of the screen. Four nav items (Today / House / Discover / Me) with a FAB "+" button in the middle for Add Plant. Active tab is highlighted. The FAB's click handler navigates to /add-plant — the actual Add Plant flow is ticket 12. Hidden on screens ≥1024px.
componentclient/src/components/Sidebar.jsx
The desktop-only fixed left sidebar. Logo, nav items with labels, "Add a plant" CTA card near the bottom, and the signed-in user's avatar/email at the foot. Visible only on screens ≥1024px (the inverse of the Dock).
pageclient/src/pages/NotFound.jsx
The 404 page. Renders for any route that doesn't match — wrong URL, typo, stale bookmark. Not much to it: a friendly "this page has wilted away" message and a link back to Today. Lives outside the protected <Layout> so it doesn't try to render the sidebar/dock frame for someone who isn't logged in.
extendclient/tests/app.spec.js
Existing Playwright smoke test — extend it with one more assertion: hitting a bogus URL renders the 404 page. No need to test the auth redirect here; ticket 5 covers that once real Login exists.

Structurally, the running app looks like this:

<App> └── <QueryClientProvider> ← TanStack Query cache (ticket 7 fills it) └── <AuthProvider> ← already built in ticket 3 └── <BrowserRouter> └── <Suspense> ← catches lazy-loaded page chunks └── <Routes> │ ├── /login → Login (placeholder for now) ├── /register → Register (placeholder) ├── /welcome → Welcome (placeholder) │ ├── <ProtectedRoute> │ └── <Layout> ← sidebar + dock │ └── <Outlet /> ← current page renders here │ ├── / → Today (placeholder) │ ├── /house → House (placeholder) │ ├── /plants/:id → PlantDetail (placeholder) │ ├── /discover → Discover (placeholder) │ ├── /me → Me (placeholder) │ └── /add-plant → AddPlant (placeholder) │ └── * → NotFound

The placeholder pages are literally a heading and a "coming soon" line — just enough to prove the routing works. Future tickets swap each one for the real thing.

Requirements (what "done" looks like)

  1. Routing is in place. Public routes exist for /login, /register, /welcome. Protected routes exist for / (Today), /house, /plants/:id, /discover, /me, /add-plant. Unknown paths render the 404 page.
  2. Auth gating works without a flash. Unauthenticated users visiting a protected route get redirected to /login. While the AuthContext is still resolving the session on initial load, the app shows a loading spinner — never the login screen briefly, then the real content.
  3. Redirect-back on login. If an unauthenticated user tries to visit /house, they get redirected to /login, and the location they were trying to reach is remembered so login can bounce them back. (You won't be able to fully test this until real Login exists in ticket 5 — but the plumbing should be there now.)
  4. Mobile and desktop each get the right frame. Viewports <1024px show the glass dock; viewports ≥1024px show the sidebar. Only one is visible at a time.
  5. Route-level code splitting. Each page is loaded on demand so the initial JS bundle only contains the shell + landing page. See the React notes below for why this matters and how React.lazy + <Suspense> fit together.
  6. 404 page. Lives outside the protected layout, matches the app's design language, has a link home.
  7. Smoke test extended. The existing Playwright test still passes, plus a new assertion that hitting a bogus URL renders the 404.

Acceptance criteria (testable in a browser)

Visiting /this-does-not-exist shows the 404 page, not a blank screen or a crash.
Visiting any protected route while logged out redirects to /login.
Refreshing the page while logged in does NOT briefly flash the login screen before showing the real content.
Clicking a nav item in the dock or sidebar changes the URL and swaps the content in the main area, without a full page reload.
Mobile viewport (<1024px): glass dock visible at bottom, sidebar not rendered.
Desktop viewport (≥1024px): sidebar visible on left, dock not rendered.
Clicking the FAB ("+") navigates to /add-plant (which shows its placeholder page for now).
DevTools → Network: navigating from / to /house fetches a new JS chunk that wasn't loaded before.
./scripts/lint.sh and ./scripts/run_tests.sh client both pass.

React concepts worth looking up before you start

React Router's layout-route pattern and <Outlet />. React Router lets you nest routes so a parent route renders "chrome" and its child routes render into a hole in that chrome. The hole is called <Outlet />. Mental model: Rails' yield inside application.html.erb. Your Layout component renders the sidebar/dock and then an <Outlet /> where the matched child route's element appears. When the URL changes from / to /house, only the Outlet's contents re-render.
The "protected route" pattern. React Router doesn't ship a built-in auth guard — you wrap a route element in a component that checks your auth context and either renders children or redirects. The tricky part is what you render while the auth state is still being determined (e.g. while the refresh-token request is in flight on boot). Three possible states: loading → spinner, no user → <Navigate to="/login" />, user → children. Pick the wrong render for "loading" and you get the login-flash bug.
React.lazy + <Suspense>. React.lazy(() => import('./pages/Today')) tells Vite to put that page in its own JS chunk. When React tries to render it, the import happens — and because it's async, React "suspends" the tree until the chunk arrives. A <Suspense fallback={...}> parent somewhere above the lazy component catches the suspension and shows your fallback UI in the meantime. Miss the Suspense boundary and you get a runtime error. Why bother? Without lazy loading, the initial bundle contains every page in the app even though the user only needs one of them on first paint. With it, each page ships separately and is fetched when someone navigates to it.
Redirect-back with router state. <Navigate to="/login" state={{ from: location }} /> stashes the current location on the navigation. Then Login can read it back via useLocation().state?.from and navigate there after successful auth. You'll fully wire this in ticket 5, but ProtectedRoute needs to set it up now.
Don't touch the AuthContext refresh flow. refreshToken inside AuthContext.jsx uses bare fetch instead of the apiGet wrapper on purpose — routing the refresh call through apiFetch would create a 401 → refresh → 401 loop. Same for refreshAccessToken in api/client.js. If something looks inconsistent there, it isn't. Leave it alone.
Tailwind breakpoints default to min-width. lg:flex means "flex at ≥1024px". For "only on mobile" elements, use the default classes and hide them with lg:hidden. For "only on desktop", use hidden lg:flex. Fewer media queries than you'd expect.

Notes & gotchas

  • Placeholder pages need to be barely more than a heading. A single PlaceholderPage({ title }) component you reuse for all six is fine — future tickets replace them one at a time.
  • If you see a "component suspended while responding to synchronous input" warning after adding React.lazy, the fix is almost always "move the <Suspense> one level up" until it wraps <Routes>.
  • The dock's FAB navigates to /add-plant. Don't try to make it open a bottom sheet now — Vaul bottom sheets are ticket 12's job.
  • SVG icons in the plan's Dock/Sidebar examples are inline and fine as-is for now. Swapping to an icon library later is trivial.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch, then open a PR.
