Requirements

  1. Greeting + date. "Good morning / afternoon / evening" based on local time, followed by the user's name. Eyebrow date like "Wed 9 April" above. On desktop, use the Fraunces italic display font for the headline (editorial moment).
  2. Conditional hero card. If any plants are in water_status === 'overdue', render a HeroCard for the single most urgent one (most negative days_until_water). Otherwise, hide the hero entirely and let the ritual list rise to the top.
  3. Since ribbon. Always rendered. Urgent variant (coral alert icon) if any tasks exist. Non-urgent variant (green check icon) if no tasks. Title and subtitle adapt to the state.
  4. Task list. Merge plants_needing_water and plants_needing_feeding from the dashboard response into a single array of tasks, each with { plant, careType }. Sort by urgency: overdue first, then due_today, then due_soon.
  5. Progress indicator. Header for the "Today's rituals" section shows "N done / M to go" and a small ProgressRing (from ticket 8). Since this ticket doesn't track "done" state on the client (logging care refetches the dashboard, which removes the plant from the list), done is always 0 — the ring reflects "tasks remaining vs. originally loaded", or just hide the count for MVP.
  6. Task row actions. Each TaskRow has a check button. Tapping it fires useLogCare(plant.id).mutate({ care_type: task.careType }). The mutation invalidates ['dashboard'], so the row disappears on the next render automatically.
  7. Empty state. If tasks.length === 0 and no hero, show a calm "no tasks today" message. If the user has zero plants total, nudge them toward adding one via the FAB.
  8. Derived state in render, not in useEffect. The task list, the urgent plant, the totals — all computed inline (or via useMemo for non-trivial work). Never sync them into state via useEffect.

Acceptance criteria

Visiting / while logged in shows the greeting with the user's name.
If the dashboard endpoint returns at least one overdue plant, the HeroCard is visible.
If all plants are healthy, the HeroCard is hidden and the ritual list is empty or near-empty.
Tapping the check on a TaskRow fires a POST /care_logs request and removes that row within a second (refetch).
The task list is ordered with overdue plants first.
If the user has zero plants, the empty state nudges them to add one.
Loading spinner appears briefly on first load, not on subsequent navigations within the same session (cache hit).
./scripts/lint.sh and ./scripts/run_tests.sh client pass.

React concepts worth looking up

useMemo for non-trivial derived arrays. Building the task list is a filter + merge + sort across the dashboard data. That's enough real work that wrapping it in useMemo(() => ..., [data]) is worth it — the sort won't re-run unless data actually changes. Compare with ticket 11, where the percentage math is trivial and shouldn't be memoised. Rule of thumb: if you'd hesitate to do this work inside a tight render loop, memoize it. If it's a one-line arithmetic expression, don't.
Module-level subcomponents, NOT inline. WaterableHeroCard and WaterableTaskRow (the small wrappers that call useLogCare per row) live at module level in Today.jsx — NOT inside the Today function body. Defining a component inside another component creates a brand-new component type on every render of the parent. React can't reconcile a "different type" with the previous render, so it unmounts and remounts the entire subtree. State resets, animations restart, focus is lost, the network request fires twice. Totally invisible until you have child state — then very confusing. This is Vercel's rerender-no-inline-components rule and it's exactly the kind of subtle "renders happen more than you think" bug that a hobbyist React era doesn't teach you.
Derive during render, don't sync via useEffect. The task list, the urgent plant, the totals — all computed inline (or via useMemo for the heavier ones). Never store these in useState and use useEffect to keep them in sync with data. That pattern was common in the React 16/17 era and is now an explicit anti-pattern in the React docs. If a value can be derived from props or other state, derive it; don't store it. The rule of thumb: ask "what's the source of truth for this value?" — if the answer is "another piece of state", you don't need a second piece of state.
TanStack Query's cache makes navigation feel instant. When you leave Today and come back within a few minutes, the cached dashboard data renders immediately (no spinner) and TanStack triggers a silent background refetch to update it. The user sees instant content followed by a quiet refresh. This is the staleTime: 1000 * 60 * 2 default we set on the QueryClient in ticket 4 doing its job. You don't need to do anything to get this — but it's worth knowing why the app feels fast (and why your loading states should be calibrated for the empty cache, not the warm one).

Forward compatibility notes

Keep the HeroCard and TaskRow composition boring. Ticket 13 (animations) will wrap these in Framer Motion components (motion.div, AnimatePresence) to add the care-animation flourish. The simpler the composition here, the easier that wrap-and-animate is later.
useLogCare per row is intentional. Each TaskRow wrapper calls its own useLogCare. This means each row has independent pending/success state, which matters once ticket 13 animates the check-circle on successful mutate. Don't consolidate to a single top-level mutation hook — you'd lose per-row loading state.
Don't track "done" tasks on the client. When the user taps water, the mutation invalidates the dashboard, the dashboard refetches, and the plant drops out of the needing-water list. No client-side "isDone" state needed. If you feel tempted to add one, stop — you're fighting the data flow.

Notes & gotchas

  • The FAB in the dock already routes to /add-plant (ticket 4). Don't duplicate it on Today.
  • If the dashboard query fails (network, auth expiry that bypassed refresh), render a simple error state with a "try again" button. Don't let the screen be blank.
  • On desktop, the Fraunces italic headline is a deliberate editorial moment — don't swap it for the body font. Design spec §3 is specific about this.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch, then open a PR.
