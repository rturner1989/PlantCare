TICKET-006-5

Password reset — forgot password + reset flow

Open Medium

Summary

The "Forgot password?" link on Login is currently a cosmetic <span> that does nothing. This ticket makes it work end-to-end: user clicks it, enters their email, gets a link by email, lands on a page that lets them choose a new password.

In one sentence: make the forgot-password link actually useful.

Why it's 06.5, not 11: this is a gap in the auth-flow work that started in tickets 3/5. The UI has the affordance (Login's "Forgot password?" text), but there's no flow behind it. Closing that gap now — before moving on to the app-proper screens in ticket 7+ — keeps the auth surface area honest.

What came before: User has has_secure_password from ticket 1, the access/refresh token flow from ticket 2, and the authenticated password-change endpoint (Profile::PasswordsController) from ticket 3. Login/Register are from ticket 5.

What this unlocks: a production-shippable auth surface. Every feature ticket from 7 onward can assume users can recover from a lost password without support intervention.

Mental model: the reset token is a one-shot bearer credential. Issue it hashed (same as the refresh-token pattern), email the raw version exactly once, consume it on use. Same shape as the existing RefreshToken model — reuse that pattern.

What you're building

Backend: a token model, two endpoints, a mailer. Frontend: two pages and a linkified Login affordance.

modelapi/app/models/password_reset_token.rb
Stores hashed reset tokens scoped to a user. Mirrors RefreshToken: generate(user) returns [raw, record], usable? predicate (unexpired + not used), consume! marks used_at. Raw tokens never touch the database.
controllerapi/app/controllers/api/v1/password_resets_controller.rb
#create takes an email, generates a token if the user exists, hands the raw token to the mailer, responds 202 Accepted with a generic "if that email exists…" message regardless of whether the user was found (no user-existence leak). #update takes { token, password, password_confirmation }, verifies/consumes the token, updates the password. Returns 200 on success, 422 on validation failure, 410 Gone on expired/used/unknown token.
mailerapi/app/mailers/password_reset_mailer.rb
Single action, reset_instructions(user, raw_token). Renders HTML + text templates. Link points at the frontend's /reset-password/:token route via the host from action_mailer.default_url_options.
pageclient/src/pages/ForgotPassword.jsx

Flow:

Login → "Forgot password?" ──→ /forgot-password │ ▼ email submit POST /api/v1/password_resets │ ▼ (always 202) success state: "check your inbox" │ │ (meanwhile, user receives email) ▼ User clicks link ──→ /reset-password/:token │ ▼ new password submit PATCH /api/v1/password_resets/:token │ ▼ success Toast + redirect to /login

Requirements

  1. Token model with hashed storage. PasswordResetToken stores only the SHA-256 digest of the raw token, never the raw value. generate(user) returns the raw token + record; find_by_raw_token(raw) looks up by digest. 24-hour expiry baked in.
  2. No user-existence leak. POST /password_resets always responds 202 Accepted with the same body whether or not the email matches a user. An attacker can't use this endpoint to enumerate registered accounts.
  3. Token is single-use. Successfully consuming a token sets used_at. A second PATCH /password_resets/:token call with the same token returns 410 Gone.
  4. Token expires. 24 hours from issue. After that, PATCH returns 410 Gone.
  5. Password rules apply on reset. Same has_secure_password + length/composition/common-password validations as registration. Show validation errors inline.
  6. Email delivered via letter_opener_web in dev. Mounted at /letter_opener so you can browse sent mails in the browser. In production the mailer config switches to real SMTP (left as a deployment concern).
  7. ForgotPassword page. One email input, submit button, success state. Same card + form shape as Login/Register. Navigates nowhere on success — stays on the page to show the confirmation.
  8. ResetPassword page. Reads :token from the URL. Two password inputs (password + confirmation) with strength bar. On success, toast + redirect to /login. On 410, render an expired-link state.
  9. Login link. The existing <span>Forgot password?</span> becomes a real <Link> (via the Action primitive with a to prop) pointing at /forgot-password.

Acceptance criteria

Clicking "Forgot password?" on Login navigates to /forgot-password.
Submitting a valid email shows the success state and opens a reset email in /letter_opener.
Submitting a nonexistent email shows the identical success state (no leak) and sends no email.
Clicking the link in the email lands on /reset-password/:token with a password form rendered.
Submitting a valid new password redirects to /login with a success toast, and the new password works on login.
Attempting to reuse the same token returns 410 and renders the expired-link state with a link back to /forgot-password.
Password validation failures (too short, too common, mismatched confirmation) render inline field errors, not a toast.
./scripts/lint.sh and ./scripts/run_tests.sh api pass. Client unit tests pass; E2E for the reset half is deliberately skipped pending a mail-inspection helper.

Security

Never store raw tokens. The database holds the SHA-256 digest. Raw tokens live only (a) in memory during create, (b) in the email the user receives, (c) in the URL they click. Lookup is by digest. This is the same pattern RefreshToken uses — follow it exactly.
Constant-time comparison on digest lookup. Postgres WHERE token_digest = ? is fine because the digest doesn't reveal the raw token — you can compare it non-constant-time without leaking anything useful.
Filter the token param from logs. Add :token to Rails.application.config.filter_parameters so the raw token never lands in production.log.
No user-existence oracle. POST /password_resets returns the same response whether the email exists or not. This is why the #create action does the lookup, conditionally creates a token, and unconditionally responds 202 — never let the response shape or status code differ based on whether the user was found.
Rate limiting deferred to prod. Rack::Attack isn't configured in dev. When the app hits production, add a throttle on this endpoint (3 requests per IP per hour, 3 per email per hour) — that's deploy-time work, not this ticket.

Forward compatibility notes

Same mailer infrastructure unlocks other emails. Welcome emails, care reminders, weekly digests — all of them will ride on the same ActionMailer + letter_opener_web setup. After this ticket, sending any email is a matter of adding a mailer class, not setting up infrastructure.
Production SMTP is a deployment concern. The dev config uses letter_opener_web; production needs real SMTP (SendGrid, Postmark, SES — pick at deploy time). Don't bake SMTP credentials into this ticket.
Rate limiting is production-only for now. Rack::Attack throttle belongs on the POST /password_resets endpoint but should land with the production deployment work, not here. Leave a comment on the controller noting the intended throttle so the future work is discoverable.

Notes & gotchas

  • Frontend host vs API host: the reset link points at the frontend (http://localhost:5173 in dev), not the API (http://localhost:3000). Set config.action_mailer.default_url_options to the frontend host so link helpers generate the right URLs.
  • letter_opener_web mounts at /letter_opener by default — keep that path so anyone coming back to this work after a gap can find it.
  • The 410 Gone handling on the frontend needs to distinguish "token expired/used/unknown" (410) from "password validation failed" (422). Different error classes, different UI treatment.
  • Skip E2E for the reset half this time. Testing it properly needs either a mail-inspection helper (letter_opener_web's mail list is not straightforward to drive from Playwright) or a dev-only debug endpoint — both out of scope for this ticket.
