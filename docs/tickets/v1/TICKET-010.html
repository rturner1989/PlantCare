Requirements

  1. Header with view-mode toggle. "House" title + pill-style toggle with three options: Rooms, List, (Greenhouse — shown as disabled/"coming soon"). Default view is Rooms.
  2. Search input. Placed near the toggle. Type to filter; results update as you type. In Rooms mode, the search is either hidden or filters by room name (design choice — keep it simple). In List mode, it filters the flat plant list by nickname or species common name.
  3. Rooms view. Grid of RoomCards (responsive: 1 column on small, 2 on medium, 3 on large). Each card shows room name, icon, plant count, and an attention badge if any of that room's plants have water_status === 'overdue'. The attention count is computed by House, not by RoomCard.
  4. List view. Flat list of plant tiles. Each tile shows a PlantAvatar, nickname, species common name, room name, and a small care-status indicator (💧 for water-due, 🌱 for feed-due). Tap → /plants/:id.
  5. Room filter. Tapping a RoomCard switches to List mode with roomId stored in state so the list is filtered to that room. A "back to all rooms" or "clear filter" affordance returns to List showing everything.
  6. Empty states. If the user has zero rooms, show a nudge toward onboarding or adding a plant. If a room has zero plants, its RoomCard shows "0 plants" but is still clickable.
  7. Loading state. Spinner while rooms/plants are loading on first visit. Cache hit on subsequent visits means no spinner.

Acceptance criteria

Visiting /house shows the Rooms grid by default.
Clicking the List pill switches to a flat list of every plant the user owns.
Typing in the search box filters the list as you type — no submit button needed.
Clicking a RoomCard switches to List mode with the list filtered to that room.
A room with at least one overdue plant shows an attention badge on its card.
Clicking any plant tile navigates to /plants/:id (which still renders a placeholder from ticket 4 — that's fine for this ticket).
Navigating away and back shows cached content immediately — no loading flicker.
./scripts/lint.sh and ./scripts/run_tests.sh client pass.

React concepts worth looking up

Local UI state vs. server state — the most useful distinction in modern React. viewMode and searchQuery are UI state: they only matter while the user is on this screen, and the source of truth is the user's interactions. They live in useState. The rooms and plants data is server state: shared across the app, cached, refetchable, possibly stale, and the source of truth is the server. It lives in TanStack Query, never in useState. Knowing which kind a value is — and putting it in the right place — is half the battle of structuring a React app well. The most common architectural smell in React codebases is mixing the two: useState seeded from a fetch in useEffect with a manual loading/error/refetch dance. Don't.
Multiple parallel queries. const { data: rooms } = useRooms(); const { data: plants } = usePlants(); fires two requests in parallel on mount. TanStack handles them independently — they don't block each other. You can render the rooms grid as soon as rooms arrive and show a loading state for the plants list separately, or wait for both. Either pattern is fine; the requests themselves are always parallel. Mental model: it's like writing two scopes at the top of a Rails controller and trusting Rails not to serialise them — except here, each query is also cached.
useMemo for derived data with multiple inputs. The roomAttention map (roomId → count of plants needing care) and the filteredPlants array are derived from rooms, plants, AND searchQuery. Wrap them in useMemo(() => ..., [rooms, plants, searchQuery]) so they only recompute when an input actually changes. Without it, every keystroke in the search box re-runs the entire filter even if only viewMode changed. Same rule from ticket 9: memoize when there's real work; don't memoize trivia.

Forward compatibility notes

Keep view-mode state open for a third option. Phase 3 adds a 3D isometric "Greenhouse" view as a third toggle option. Your viewMode state should be a string (not a boolean), and the toggle should render a map over an array of modes — so adding a new mode is a one-line change, not a rewrite.
Room filter is client-side, not a query param. When the user taps a room to enter the filtered list, keep the filter in local state (filteredRoomId). Don't push it to the URL as ?room_id=X — that would break the "back button returns to Rooms grid" behaviour. (Later, you could add URL state for deep linking, but not now.)
Plant tile click uses useNavigate, not <Link>. Or both work, but pick one. If you use <Link>, you lose the ability to handle card-level clicks vs. button-inside-card clicks cleanly. useNavigate inside an onClick is more flexible for compound cards.
The List view's plant tile is the fastest path to PlantDetail. Ticket 11 depends on this navigation working. The placeholder page from ticket 4 is still mounted at /plants/:id, so clicking will show that placeholder until ticket 11 swaps it. That's expected behaviour.

Notes & gotchas

  • The Greenhouse toggle option should be visibly disabled ("Coming Phase 3") rather than hidden — it sets user expectations and reinforces the design spec's three-modes IA.
  • Don't debounce the search input for this ticket. Client-side filter, instant results, no API call per keystroke. The only debounced search in the app is species search (onboarding + AddPlant), which hits the API.
  • If useRooms or usePlants errors (e.g. auth expired mid-session), render a simple error state — don't let the screen be blank.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch, then open a PR.
