TICKET-003

Auth Context + useAuth Hook

Open High Priority

Summary

Create the React auth context that manages user session state — login, register, logout, and automatic session restoration on page load. Wraps the API client from TICKET-002 into a React-friendly provider pattern. Every component that needs to know "who is logged in?" will consume this context via the useAuth() hook.

Requirements

  1. Create AuthContext with a provider that exposes: user, loading, error, login(), register(), logout(), refreshToken(), setError()
  2. On mount, attempt session restoration by calling POST /api/v1/token (refresh cookie). If successful, fetch user profile and populate user state
  3. login(email, password) calls POST /api/v1/session, stores the access token via setAccessToken(), and sets user state from the response
  4. register(name, email, password, passwordConfirmation) calls POST /api/v1/registration, same token/user flow as login
  5. logout() calls DELETE /api/v1/session, clears access token and user state. Gracefully handles already-expired tokens
  6. loading is true during initial session restoration — components should show a loading state, not a flash of the login page
  7. Create useAuth() hook that throws a helpful error if used outside the provider
  8. Memoize the context value to prevent unnecessary re-renders of consuming components

Acceptance Criteria

client/src/context/AuthContext.jsx exists and exports AuthContext and AuthProvider
client/src/hooks/useAuth.js exists and exports useAuth
useAuth() throws "useAuth must be used within an AuthProvider" when called outside the provider
AuthProvider calls POST /api/v1/token on mount to restore session from refresh cookie
login() sets access token via the API client's setAccessToken and populates user state
register() sets access token and populates user state
logout() calls DELETE /api/v1/session, then clears both token and user — even if the DELETE fails
loading starts as true and becomes false after mount refresh resolves (success or failure)
Context value is wrapped in useMemo with appropriate dependency array
./scripts/lint.sh passes with zero errors

Design References

Technical Spec — Auth Flow

Access token: 15-minute expiry, stored in memory. Sent via Authorization: Bearer header.

Refresh token: 30-day expiry, httpOnly cookie. Browser sends automatically with credentials: 'include'.

Session restore: On page load, POST to /api/v1/token. If 200, use the new access token to GET /api/v1/profile and hydrate user state. If 401, user is logged out.

API Endpoints Used

POST /api/v1/session — body: { session: { email, password } } — returns: { access_token, user }

POST /api/v1/registration — body: { user: { name, email, password, password_confirmation } } — returns: { access_token, user }

DELETE /api/v1/session — no body — returns: 204

POST /api/v1/token — no body (refresh cookie sent automatically) — returns: { access_token }

GET /api/v1/profile — returns: { id, email, name, timezone }

UI Spec §6 — Auth Screens (context for how this is consumed)

Login and register are single-screen forms. The loading state from AuthContext determines whether to show a spinner or redirect to the login page — avoids the "flash of login screen" on refresh.

Notes

  • The plan includes the complete implementation for both files in Steps 3.1 and 3.2 — use them as the reference.
  • The refreshToken function in the context makes a direct fetch() call (not apiFetch) because it needs to bypass the API client's own refresh logic — otherwise you'd get infinite recursion.
  • After a successful refresh, the context fetches GET /api/v1/profile to get the user object. The login/register endpoints return the user directly, so no extra call needed there.
  • useCallback on all action functions prevents unnecessary effect re-fires in consuming components.
  • The AuthProvider will be wired into the component tree in Task 4 (Router + App Shell). For now, just create the files — they don't need to be imported anywhere yet.
  • Error state (setError) is exposed so forms can clear errors when the user starts typing again.
