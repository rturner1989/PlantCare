TICKET-015

Rack::Attack throttle for password reset endpoint

Open High

Summary

The POST /api/v1/password_resets endpoint is unthrottled. An attacker can hammer it to enumerate which email addresses belong to real users (by timing differences, outbound mail bursts, or any future error-shape regression), and can cheaply spam real users' inboxes with reset emails they didn't ask for. A TODO(security) in password_resets_controller.rb:14 has flagged this since the endpoint landed.

In one sentence: make the reset endpoint boring to attack.

Where this fits

Phase 1 launch-readiness. The endpoint itself is correct — constant-time, doesn't leak via response body — but lacks per-IP and per-email rate caps. Rack::Attack is already a documented part of the project's security stance (see the global CLAUDE.md's Security section), so this is a wire up, not a design.

Scope note. This ticket is the password reset endpoint specifically. Broader throttling (login, register, token refresh) deserves its own ticket and audit — don't scope-creep here.

What you're building

configapi/config/initializers/rack_attack.rb
New initializer. Two throttle rules for POST /api/v1/password_resets: one keyed by remote IP (e.g. 10 req / hour), one keyed by the email in the body (e.g. 3 req / hour). Responds with 429 Too Many Requests. Rack::Attack's default in-memory cache is fine for dev; prod should use the existing Redis cache store (ActiveSupport::Cache::RedisCacheStore on db 2).
configapi/config/application.rb
Require rack/attack and insert it into the middleware stack. Standard one-liner.
testapi/test/controllers/api/v1/password_resets_controller_test.rb
Add two tests: the 11th request from a single IP returns 429; the 4th request for a single email (from different IPs) returns 429. Reset the Rack::Attack cache in setup so tests don't cross-contaminate.

Requirements

  1. Per-IP throttle on POST /api/v1/password_resets: 10 requests per hour. Configurable via a constant near the top of the initializer.
  2. Per-email throttle on the same endpoint: 3 requests per hour. Key is the downcased email from the request body.
  3. Over-limit requests return 429 Too Many Requests with a small JSON body ({ error: "Too many requests, please try again later." }), not the default Rack::Attack response.
  4. Rack::Attack uses the same Redis (db 2) as Rails.cache in prod so throttle counters survive a Puma restart and are shared across workers.
  5. The TODO(security) comment in password_resets_controller.rb is deleted (it self-resolves).
  6. No throttle on other endpoints touched in this ticket — login/register/token stay as-is until their own ticket.

Acceptance criteria

From a single IP, the 11th POST to the password reset endpoint within an hour returns 429. Requests 1–10 still return the usual 200 + email dispatch.
From three different IPs posting the same email, the 4th request returns 429 regardless of IP.
The 429 body is the project's JSON shape, not Rack::Attack's default text.
Dev: the initializer loads without errors on docker compose up; a docker compose exec redis redis-cli -n 2 KEYS "rack_attack:*" shows keys after a handful of requests.
Brakeman and bundler-audit both remain clean (./scripts/lint.sh).

Notes & gotchas

  • Don't lift and shift a generic Rack::Attack config. Most tutorial configs throttle everything. We want the narrowest rule-set that covers this one endpoint so that future broader throttling can land as a standalone ticket without conflict.
  • Email-key throttle needs body access. Rack::Attack's throttle block gets the request object; parse the body carefully (content-type is JSON). Don't mutate the request — use req.body.rewind after reading.
  • Tests must reset throttle state. Rack::Attack.cache.store.clear in the setup block, or use Rack::Attack.reset! if Rails version supports it.
  • Don't lower the limits to test-friendly numbers in prod. Use test-only overrides via env var or constants override if you need quicker limits in the test path.
