TICKET-011

Data-driven popular species rankings

Open Medium

Summary

Replace the hardcoded Species::POPULAR_NAMES list with a nightly Sidekiq job that rebuilds popularity from real user adoption — ranking species by distinct users who own one, not total plant count. Below a configurable volume threshold, the job keeps the editorial list as a fallback so cold-start onboarding doesn't see a near-empty list or a list skewed by the first three signups.

In one sentence: let the users decide which species are "popular", but don't trust them until there are enough of them.

What came before: ticket 6 shipped the onboarding wizard with a "popular picks" empty state on Step 3 — the chips a user sees before typing into species search. It reads species.popular = true, which today is set from a hand-curated list in Species::POPULAR_NAMES.

What this unlocks: launch-readiness for the onboarding wizard. As soon as the product has a few hundred users, the empty state starts reflecting what people actually grow instead of your taste. It also gives you one test-bed Sidekiq job, which will come up again in Phase 2 (reminders / adaptive scheduling).

Parallel track: tickets 7–10 are a frontend sequence. This one is backend-only — it can slot in at any point while the frontend work is in flight. It does not block and is not blocked by any of them.

Mental model: a counter_cache you compute on a schedule instead of on every insert. Same idea — denormalise a count onto the parent row — except the "count" is "distinct users", the refresh is batched overnight, and there's a cold-start fallback so the count is allowed to be unreliable while the dataset is small.

What you're building

A Sidekiq job, a scheduler wiring, a pair of small Species-model additions, a test, and a one-line change to the seed so the editorial fallback is always present as a starting state.

jobapi/app/jobs/recalculate_species_popularity_job.rb
The nightly job. Counts distinct users per species by joining through Plant → Room → User. If the total signal (number of users who own any plant) is below a threshold, leaves the editorial fallback intact and exits. Otherwise, flips the popular flag for the top N species by distinct-user count and clears it on the rest — except editorial species are still allowed to stay "popular" if they'd otherwise drop out, so the empty state never falls below a floor count. Idempotent: re-running mid-day produces the same result as the 3am run.
configapi/config/schedule.yml
The Sidekiq schedule. One entry: run the job every night at ~3am server time. Wired via sidekiq-cron (or the scheduler of your choice; see Notes). Held in YAML so the cadence is reviewable without grepping Ruby.
extendapi/app/models/species.rb
Add a class method the job calls — something like Species.distinct_user_counts that returns a hash of {species_id => user_count}. Keep the POPULAR_NAMES constant around as the editorial fallback. The existing scope :popular and the popular boolean column stay untouched — the controller doesn't care how the flag got set.
migrationapi/db/migrate/YYYYMMDD_add_popularity_score_to_species.rb
Optional but recommended: add an integer popularity_score column (nullable, default nil) so the job writes the distinct-user count alongside the boolean. Useful for debugging ("why is this plant popular?") and lets you rank within the popular set later without another computation. If you'd rather avoid the column for now, skip this and the job only flips the boolean — fine for MVP, just less legible when the data gets interesting.
testapi/test/jobs/recalculate_species_popularity_job_test.rb
Three scenarios: (1) below threshold → editorial list is preserved, nothing else changes; (2) above threshold → top-N by distinct users win; (3) above threshold but an editorial species would drop out → it stays in. The distinct-users-not-plants rule needs its own assertion: one user with 10 pothos should count once, not ten times.
extendapi/db/seeds/species.rb
No behavioural change — just confirm that the seed still sets popular: true on POPULAR_NAMES matches so a fresh DB has an editorial state the job can preserve. (It already does this today; this ticket just makes the behaviour load-bearing.)

Requirements

  1. Ranking signal is distinct users, not plant count. A user with 40 pothos counts as one vote for Pothos. A species with 40 users owning one plant each outranks a species with 1 user owning 40 plants.
  2. Cold-start fallback. Below a minimum-volume threshold (configurable; start around 50 distinct plant-owning users app-wide), the job preserves the existing editorial flags and exits without changes. Above the threshold, data-driven ranking kicks in.
  3. Editorial floor. Even once data-driven mode is active, the onboarding empty state must never drop below a sensible count (≥ 6 species). If the data-driven top-N would leave an editorial species out and would leave fewer than that floor in total, that species keeps its popular flag.
  4. Idempotent. Running the job twice in a row produces the same DB state as running it once. Safe to trigger manually from the Rails console without ordering worries.
  5. Scheduled nightly. Runs automatically via Sidekiq scheduling at an off-peak hour. The schedule is declarative (YAML or an initializer), not wired inline in random Ruby.
  6. Observable. The job logs a one-line summary when it runs: threshold hit or not, how many flags flipped, how many editorial fallbacks retained. No dashboard needed — just readable logs.
  7. No controller change. Species::PopularController#index (or wherever Species.popular is read today) does not need to know data-driven mode is active — the flag is the only contract.

Acceptance criteria

With a fresh seeded DB (zero users), Species.popular.count equals the editorial list size — the job, if triggered, leaves it alone.
After seeding ≥ threshold users with varied plant ownership, running the job updates the popular flag to reflect top-N by distinct-user count.
A user with many plants of one species contributes exactly one vote for that species.
Running the job twice produces identical results — no oscillation, no duplicate work.
The Sidekiq Web UI at /sidekiq shows the scheduled job entry with the next-run timestamp.
The onboarding wizard's /api/v1/species/popular response is still structurally identical — just the contents of the list have shifted to match usage.
./scripts/lint.sh and ./scripts/run_tests.sh api pass.

Concepts worth a quick skim

Why nightly, not per-write. The obvious alternative is keeping popular live — recompute on every plant insert/delete. Don't. Distinct-user-per-species is a non-trivial aggregation, every plant mutation would trigger it, and the result changes meaningfully maybe twice a week once you have real volume. Batched overnight is the right shape; this is the classic "precompute on a schedule" pattern.
Sidekiq scheduling options. Sidekiq itself doesn't schedule — it runs jobs on demand. For recurring jobs the two standard picks are sidekiq-cron (YAML schedule, reads on boot) and sidekiq-scheduler (similar, slightly different API). Either is fine. If you want to avoid a new dependency, Rails 8's solid_queue has built-in recurring tasks — but switching queue backends for one job is a bigger change than adding one gem. Recommendation: sidekiq-cron, since you're already on Sidekiq.
Distinct-count SQL patterns. Rails .distinct.count(:user_id) on a joined relation generates COUNT(DISTINCT plants.user_id) per group — exactly what you want here. Just watch for the classic trap: .joins(:room).group('species.id').distinct.count('rooms.user_id') is correct, but stacking .distinct before .group sometimes produces subtly different SQL. If the test case for "one user with 10 plants counts once" passes, the query is right.
Fallback logic belongs in the job, not the reader. The Species::PopularController stays dumb — it queries where(popular: true). All "editorial vs data-driven" logic lives inside the job, where it's easy to test and easy to change. This is the usual DHH-style move: controllers stay thin; domain rules live somewhere you can read them in one place.

Notes & gotchas

  • Don't remove POPULAR_NAMES. It's the editorial fallback, and the seed reads it. The job needs to know about the editorial list to implement the floor rule. Demoting it to a private constant inside the job is fine; deleting it is not.
  • Threshold values are configuration, not magic numbers. Put the "minimum volume" and "top N" values as constants at the top of the job file (or in an initializer). You'll want to tune these once real traffic lands.
  • Test data considerations. Minitest fixtures get awkward when you want "50 users, each with 3 plants across 2 rooms". A small helper that builds the graph programmatically inside the test may beat hand-writing fixture YAML. Use judgement.
  • Scope reminder: this ticket does not expose popularity data to the client (no new API fields, no sorting by score in responses). The onboarding wizard just sees the same popular: true contract it sees today — the list behind that flag is what changes.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh api, commit on a feature branch, then open a PR.
