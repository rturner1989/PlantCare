TICKET-012

URL-based wizard steps for onboarding

Open Medium

Summary

The onboarding wizard's step state lives entirely in React useState. The URL stays at /welcome through every step, so the browser's history has no idea the user has advanced. Two-finger swipe-back on macOS (or hitting the back button) on Step 3 drops the user straight to Step 1 — because as far as the browser's concerned, Step 3 was never a navigation event.

Fix by making each step its own route: /welcome, /welcome/rooms, /welcome/species, /welcome/environment, /welcome/done. Advancing becomes a navigate() call; the browser's back/forward buttons and swipe gestures work naturally as a result.

In one sentence: move wizard progression from component state into the URL so browser history tracks it.

What came before: TICKET-006 shipped the onboarding wizard end-to-end with in-memory step state. It works for the forward flow, but it never represents wizard progression as browser history — an oversight that's invisible until a user uses a native back gesture mid-flow.

What this unlocks: nothing is blocked on this, but it's worth fixing before real users touch onboarding. Once each step has its own URL, three bonus wins fall out: refresh preserves position without depending on server-side resume logic; the "resume mid-wizard" E2E test can replace its heuristic (look for server-persisted rooms, jump to Step 3) with a real URL-based restore; and analytics can measure drop-off per step cleanly.

Mental model: this is the React equivalent of converting a Rails controller that swaps partials via Turbo frames into one that uses proper REST routes for each step. The UX looks identical; the difference is that the browser's history now tracks it.

Option A (single route, :step? param) keeps the refactor tight — one component, one route. Option B (nested <Route> children with <Outlet />) is more idiomatic and opens up per-step lazy loading later, but adds route-table churn. Start with A unless you're craving the nesting.

Requirements

  1. Each step has a distinct URL. The five step slugs are stable — they're part of the app's URL contract and shouldn't change casually.
  2. Browser back/forward work through the wizard. Back from Step 3 lands on Step 2 with its previous state (the selected rooms). Forward from Step 2 returns to Step 3. Swipe gestures on macOS behave identically.
  3. Refresh preserves position. Hitting refresh on /welcome/species lands the user back on Step 3 — no "resume from server data" heuristic needed. If the user refreshes mid-wizard without having persisted anything yet (e.g. refresh on Step 3 but no rooms were saved), the app falls back to Step 1 cleanly without crashing.
  4. Direct-linking to a step is valid. Opening /welcome/species directly (say, from a bookmark) renders Step 3. If the user hasn't completed prerequisite steps (no rooms yet), the step component handles that gracefully — same as the existing Step 3 behaviour when availableRooms is empty.
  5. Unknown step slugs redirect to Step 1. A URL like /welcome/banana lands the user on /welcome, not a crash or a 404.
  6. The "onboarded user visits /welcome" guard still works. ProtectedRoute bounces onboarded users to / regardless of which step slug they hit.

Acceptance criteria

Advancing through the wizard updates the URL at each step: /welcome → /welcome/rooms → /welcome/species → /welcome/environment → /welcome/done.
Two-finger swipe back in Safari on Step 3 lands on Step 2, not Step 1. Same for the back button and history.back() in DevTools.
Refreshing on /welcome/species renders Step 3, not Step 1.
Typing /welcome/not-a-step into the URL bar redirects to /welcome without a crash or a blank screen.
The existing "un-onboarded user logging in lands at /welcome" E2E test still passes — no regression in the auth→onboarding handoff.
The full happy path E2E (register → wizard → dashboard) still passes. Any step-transition assertions are updated to also check the URL.
./scripts/lint.sh and ./scripts/run_tests.sh client pass.

React concepts worth looking up

State in the URL vs state in the component. The big idea here: some state belongs in the URL (which page am I on, which filter am I applying, which item am I viewing) and some belongs in the component (is this dropdown open, what's half-typed in this input). URL state is shareable, refreshable, and bookmarkable. Component state is ephemeral. Wizard progression is classic URL state — users expect back/forward to work and refresh to land them where they were. Your current wizard got the call wrong; this ticket corrects it.
useParams(). Returns the URL parameters React Router parsed from the current route. Given a route like <Route path="/welcome/:step?" ... />, const { step } = useParams() gives you the current slug (or undefined when the user's on bare /welcome). Pure function of the URL — no subscription plumbing, no effect needed.
useNavigate() vs <Link>. You already use both elsewhere. <Link> is for user-initiated navigation (an anchor they can right-click, command-click, etc.). useNavigate() is for programmatic navigation from event handlers — *after* the rooms mutation succeeds, navigate to /welcome/species. Both push a new entry onto the browser's history stack, which is exactly what makes back/forward work.
Resume logic can probably go away. The current hasResumedRef effect in Welcome.jsx exists to handle "user refreshes mid-wizard" by looking at server-persisted rooms and jumping forward. Once the step is in the URL, a refresh on /welcome/species just… lands on Step 3. The server resume heuristic becomes unnecessary for this specific use case. You may still want a guard for the "landed on Step 3 but availableRooms is empty" case — but that's a data-level check, not a state-restoration hack.
One route or nested routes? React Router v7 supports both. <Route path="/welcome/:step?" ... /> is the simplest — one Welcome component handles all five steps internally. <Route path="/welcome" element={<Welcome />}><Route path="rooms" element={<Step2 />} />...</Route> with an <Outlet /> inside Welcome is the idiomatic v7 way — each step can later code-split independently, have its own layout, its own loader. For a 5-step wizard with shared chrome, nested is overkill. Start with the param.

Notes & gotchas

  • Don't touch the step components. Step1Intro, Step2Rooms, Step3Species, Step4Environment, Step5Done all keep the exact same props interface — onBack, onComplete, initial data. The Welcome shell is what changes: it passes () => navigate(-1) for onBack and () => navigate('/welcome/species') for onComplete instead of setStep wrappers.
  • Step 2's case-duplicate E2E test has a known assertion on "stay on Step 2 after error". The existing test hits the backend and expects to see Step 2 still rendered. With URL-based routing, "still on Step 2" means /welcome/rooms in the URL plus the Step 2 UI visible. Update that assertion.
  • Watch for the useEffect that bounces onboarded users to /. It's keyed on user?.onboarded and should still fire regardless of which step slug the user lands on. ProtectedRoute handles the outer guard; this effect is belt-and-braces.
  • Step state that isn't step progression (the wizard's data: created rooms, selected species, nickname) still belongs in useState. Don't be tempted to cram it into the URL too — it's ephemeral, not shareable, and would bloat the URL bar. URL tracks where; local state tracks what.
  • The hasResumedRef server-resume effect is probably dead code after this. Read it carefully before deleting — it handles the edge case of "user started onboarding, closed the tab, came back later". With URL-based steps, they come back to wherever they bookmarked, but if they didn't bookmark and just re-open /welcome, they land on Step 1 by default. Decide whether to keep the "jump to Step 3 if rooms exist" behaviour or drop it. Either answer is fine; be deliberate.
