TICKET-002

API Client (fetch wrapper)

Open High Priority

Summary

Create a custom fetch wrapper that handles JWT authentication, automatic token refresh on 401 responses, and request queueing during refresh. This is the foundational API layer — every protected API call flows through this module. Access tokens are stored in a JavaScript closure (never localStorage), and the refresh token lives in an httpOnly cookie managed by the browser.

Requirements

  1. Create client/src/api/client.js exporting: apiFetch, apiGet, apiPost, apiPatch, apiDelete, setAccessToken, getAccessToken
  2. Store access token in a module-scoped closure variable — never in localStorage, sessionStorage, or cookies
  3. Automatically attach Authorization: Bearer <token> header to all requests when a token exists
  4. On 401 response, attempt token refresh via POST /api/v1/token (credentials: include), then retry the original request once
  5. Queue concurrent requests during a refresh — don't fire multiple refresh calls simultaneously
  6. Handle FormData bodies (for photo uploads) by removing Content-Type header so the browser sets the multipart boundary
  7. Return null for 204 No Content responses, parsed JSON for all others
  8. Throw structured errors with status and body properties on non-OK responses
  9. Throw "Session expired" error when refresh fails (caller handles redirect to login)

Acceptance Criteria

client/src/api/client.js exists and exports all 7 functions listed in requirement 1
Access token is stored in a module-scoped let variable — no calls to localStorage or sessionStorage anywhere in the file
apiFetch attaches Authorization: Bearer header when token is set
On 401, the wrapper calls POST /api/v1/token with credentials: 'include' and retries the original request with the new token
Concurrent 401s during refresh are queued (not duplicated) — isRefreshing flag + refreshQueue array pattern
apiPost with a FormData body does NOT set Content-Type: application/json
Failed requests throw an Error with .status and .body properties
./scripts/lint.sh passes with zero errors
./scripts/run_tests.sh client passes (app still loads without import errors)

Design References

Technical Spec — Auth Strategy

Access token: 15-minute expiry, stored in memory on the client. Sent via Authorization: Bearer header.

Refresh token: 30-day expiry, stored as an httpOnly cookie. Used to obtain new access tokens.

This means the client never directly handles the refresh token — it's sent automatically by the browser with credentials: 'include'.

API Endpoints Used by This Module

POST /api/v1/token — exchange refresh cookie for a new access token. Returns { access_token }.

All other endpoints use Authorization: Bearer <access_token> for authentication.
