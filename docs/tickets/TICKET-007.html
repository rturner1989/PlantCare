TICKET-007

TanStack Query hooks — the data layer

Open Medium

Summary

Wrap every REST endpoint the app will call in a TanStack Query hook. This ticket produces no visible UI — it builds the plumbing that every screen from ticket 9 onwards will consume. When it's done, a component can write const { data, isLoading } = useRooms() and get a cached, auto-refetched list of the user's rooms for free.

In one sentence: convert the raw apiGet/apiPost wrapper into a library of cacheable, auto-invalidating hooks.

What came before: ticket 2 gave you api/client.js — a raw fetch wrapper that handles JWT + refresh. It's fine for one-shot calls (onboarding uses it) but it doesn't cache, dedupe, or refetch on invalidation. Ticket 4 already mounted QueryClientProvider in App.jsx so useQuery and useMutation have a cache to attach to.

What this unlocks: every single screen ticket from here on. Today (9), House (10), PlantDetail (11), AddPlant (12), Me (14) — they all consume these hooks. This is the glue that makes the screens feel instant.

Mental model: this is the React equivalent of a set of Rails controller actions plus ActiveRecord relations plus Rails.cache, all rolled together. A useQuery hook is a controller action that fetches, caches the result keyed by its arguments, and automatically hands the data to the caller.

What you're building

Five hook files, one per domain. No UI — every file exports named functions that return { data, isLoading, error } (for queries) or { mutate, isPending } (for mutations).

hookclient/src/hooks/useDashboard.js
Single read hook wrapping GET /api/v1/dashboard. Returns plants needing water, plants needing feeding, upcoming care, and stats. The Today screen (ticket 9) is the one consumer.
hookclient/src/hooks/useRooms.js
Read + write hooks for the rooms resource: useRooms(), useRoom(id), useCreateRoom(), useUpdateRoom(), useDeleteRoom(). Mutations invalidate the ['rooms'] query key so the UI auto-refreshes. The House screen (ticket 10) and onboarding (ticket 6, although onboarding uses bare apiPost instead) consume these.
hookclient/src/hooks/usePlants.js
The biggest file. Read hooks: usePlants(roomId), usePlant(id), useCareLogs(plantId), usePlantPhotos(plantId). Mutation hooks: useCreatePlant(), useUpdatePlant(), useDeletePlant(), useLogCare(plantId), useUploadPhoto(plantId), useDeletePhoto(plantId). Mutations invalidate the right keys: logging care invalidates ['plants', id], ['plants'], ['careLogs', plantId], AND ['dashboard'] (because the dashboard might change).
hookclient/src/hooks/useSpecies.js
useSpeciesSearch(query) with a longer staleTime (5 minutes — species data barely changes) and enabled: query.length >= 2 to avoid firing with empty input. Also useSpecies(id) for loading one species by id.
hookclient/src/hooks/useProfile.js
useProfile(), useUpdateProfile(), useChangePassword(). Mutation invalidates ['profile']. The Me screen (ticket 14) is the consumer.

Requirements

  1. Every API endpoint has a hook. Compare your hooks against the "API Reference" section at the top of the frontend plan. Every endpoint in that list must map to one of the hooks in these files.
  2. Consistent query key shape. Resource-level queries use [resourceName] (e.g. ['rooms'], ['plants']). Item-level queries use [resourceName, id] (e.g. ['plants', 42]). Scoped queries use [parent, parentId, child] (e.g. ['careLogs', plantId]). Inconsistency here will bite you during invalidation.
  3. Mutations invalidate the right keys. Creating a room invalidates ['rooms']. Logging care invalidates ['careLogs', plantId], ['plants', plantId], ['plants'], AND ['dashboard']. Think about what queries would go stale when this mutation succeeds and invalidate them all.
  4. Conditional fetching via enabled. usePlant(id) should enabled: !!id so it doesn't fire with undefined. useSpeciesSearch(q) should enabled: q.length >= 2 so single-character searches don't hit the API.
  5. usePlants accepts a primitive roomId, not an object. The plan was previously passing params = {} as a default which burns a new object on every render; the corrected signature is usePlants(roomId). Follow the corrected shape.
  6. No UI in this ticket. Don't build a throwaway test page. The hooks get exercised by ticket 9 (Today) immediately after this one.

React concepts worth looking up

Why hooks instead of useEffect + fetch. The 2021 pattern you probably remember: useEffect(() => { setLoading(true); fetch(url).then(setData).catch(setError).finally(...); }, [...]), plus a manual abort controller, plus a cleanup function, plus duplicate logic in every component that needs the same data. TanStack Query handles all of that — plus caching, deduplication, refetch-on-focus, retries, background updates, and pagination — for free. If you find yourself reaching for useEffect + fetch in this codebase, stop and reach for useQuery. The only place that still uses bare apiGet/apiPost is onboarding (ticket 6), where one-shot writes don't benefit from a cache.
useQuery. Takes a queryKey (an array that uniquely identifies this fetch) and a queryFn (an async function that fetches the data). Returns { data, isLoading, error, refetch }. TanStack handles caching, deduplication, and background refetching automatically. Two components calling useQuery({ queryKey: ['rooms'] }) only fire one network request — TanStack dedupes by key.
useMutation. Takes a mutationFn and optional onSuccess/onError. Returns { mutate, mutateAsync, isPending, isError }. You call mutate(payload) from an event handler. Mutations don't automatically invalidate anything — you have to tell them what to invalidate in onSuccess. This is the most common new-to-TanStack confusion: the cache doesn't magically know your POST changed the rooms list, you have to call queryClient.invalidateQueries({ queryKey: ['rooms'] }) yourself.
Query keys are arrays — and they form a hierarchy. ['plants'] is the "all plants" query. ['plants', 42] is "plant 42". ['plants', 42, 'photos'] is "plant 42's photos". TanStack treats these as a tree for invalidation: calling invalidateQueries({ queryKey: ['plants'] }) also invalidates ['plants', 42] and ['plants', 42, 'photos']. Design your keys with this in mind so a single invalidate call cascades correctly — and so a misplaced extra key (like a stray object literal) doesn't accidentally split a query into two cache entries that never invalidate together.
staleTime vs gcTime. staleTime is how long TanStack considers the cached data "fresh" (no refetch needed). gcTime is how long it keeps the data in memory after the last component unsubscribes. The QueryClient default we set in ticket 4 (staleTime: 1000 * 60 * 2) is fine for most things. Species search gets a longer staleTime because species data barely changes between sessions. Don't reach for these unless you have a specific freshness requirement.

Forward compatibility notes

Consistent query keys are a forward-compat contract. If ticket 9 invalidates ['plants'] after logging care, it's relying on every plants query using that key shape. If you introduce ['plantList'] somewhere, you'll have a ghost query that never refreshes. Grep for queryKey: before finishing to make sure nothing's off-shape.
useLogCare must invalidate ['dashboard']. Ticket 9's Today screen reads from useDashboard(). Logging care changes what Today shows — if you don't invalidate the dashboard key, the task row will stay rendered even after the user taps "done". The plan has the full invalidation list; match it exactly.
Ticket 12 (Add Plant) will call useSpeciesSearch with useDeferredValue. This ticket's responsibility is just to expose useSpeciesSearch(query) — the debouncing is ticket 12's job. Don't bake a debounce into the hook; it's the caller's decision.

Notes & gotchas

  • Don't bother testing these hooks in isolation. They're thin wrappers around apiGet/apiPost; ticket 9 will exercise them for real immediately.
  • If you see a "useQueryClient must be used within a QueryClientProvider" error, something's importing a hook from outside the provider tree — double-check ticket 4's App.jsx wraps everything in <QueryClientProvider>.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch, then open a PR.
