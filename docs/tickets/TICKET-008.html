What you're building

Six reusable components, one utility, and two data files for the personality system. None of them fetch data — they're all props-in, JSX-out.

componentclient/src/components/ProgressRing.jsx
SVG circular progress indicator. Props: value (0–100), size, strokeWidth, color, children (centred content, e.g. an emoji or a number). Used on Today (task progress), PlantDetail (water/feed/mood rings), and potentially Me (streak).
componentclient/src/components/PlantAvatar.jsx
Small illustrated plant tile. Props: species, size. Uses a lookup on species.common_name or a fallback to a generic pot. Phase 3 will replace this with a real illustration library — for MVP, coloured SVG or emoji is fine.
componentclient/src/components/TaskRow.jsx
A single care task row: plant avatar, plant name, voice quote, care type icon, and a check button. Props: plant, careType ('watering' | 'feeding'), voiceQuote, done, onComplete. When done is true, the check button renders as a filled tick.
componentclient/src/components/HeroCard.jsx
The forest-gradient urgent plant hero. Props: plant, onWater. Shows the plant's nickname, a personality-driven speech quote, room + overdue days, and a "Water <name> →" CTA. This is the signature visual moment of the app — treat it with care.
componentclient/src/components/SinceRibbon.jsx
The compact since-you-were-gone ribbon. Props: urgent (boolean), title, subtitle. Mint background with a coral alert icon if urgent, green checkmark if not. Used at the top of Today.
componentclient/src/components/RoomCard.jsx
Room tile for the House grid. Props: room, attentionCount, onClick. Shows room name, icon, plant count, and an attention badge (coral dot + count) if any plants need care. Clicking invokes the onClick prop.
utilclient/src/utils/careStatus.js
Pure function: getCareState(plant) returns one of thriving, content, thirsty, parched, dying based on water_status, feed_status, and days_until_water. Also exports getDaysDisplay(days) for human-readable "8 days overdue" / "due in 3 days" etc.
dataclient/src/personality/states.js
Config map from care state to visual values. CARE_STATE_CONFIG = { thriving: { emoji, label, color }, ... }. Referenced by every component that renders state.
dataclient/src/personality/voices.js
The voice line library and getVoiceLine(personality, state, plantId) selector. Exports a lookup table keyed by personality (dramatic, prickly, chill, needy, stoic) then by care state, with an array of quote strings each. The selector uses the plant id as a deterministic seed so the same plant keeps saying the same line across renders — no "jumping voice lines" when components re-render.

Requirements

  1. Every component renders standalone. Given only its props, each component produces valid JSX without needing a parent provider or a specific context.
  2. No data fetching inside components. Props in, JSX out. If a component needs plant data, it takes a plant prop — it does not call usePlant internally. This is so they can be composed inside different parents without duplicating queries.
  3. Care state covers all five levels. getCareState must be exhaustive — every possible water_status/feed_status combination should return one of the five defined states. No undefined.
  4. Voice selection is deterministic per plant. Given the same (personality, state, plantId), getVoiceLine must return the same quote. Use something like quotes[plantId % quotes.length] so re-renders don't cause the voice to flicker between different lines.
  5. Personality lookup handles unknown personalities. If a species has a personality string we haven't written lines for, fall back to chill rather than crashing.
  6. Visuals match the mockups. Hero card uses the forest→forest-2 gradient. Progress rings use a smooth SVG arc, not a CSS conic-gradient hack. RoomCard matches mockup 05's tile layout.

Acceptance criteria

Every component in the artifact list exists and exports a default function component.
None of the component files import from @tanstack/react-query — they're pure presentation.
getCareState returns a valid state string for a plant with every combination of water_status and feed_status values in the API.
getVoiceLine(personality, state, plantId) returns the same string on repeat calls with the same arguments (test it in the Rails console for the React equivalent — a browser console.log works).
./scripts/lint.sh and ./scripts/run_tests.sh client pass.
At least one component is imported and rendered somewhere to verify it works in context (a throwaway /playground route is fine, or just wait for ticket 9 which will use them).

Design spec §6 (personality system):

Species have a personality attribute (dramatic, prickly, chill, needy, stoic) that drives voice lines per care state. Voice lines are editorial — short, character-driven quotes that feel like the plant is speaking. A Monstera with personality: dramatic in state parched might say "I'm literally dying in here." A cactus with personality: prickly in the same state says "I've had worse."

React concepts worth looking up

Pure components: props in, JSX out — and why it matters here. Every component in this ticket takes data via props and returns JSX. None of them call useQuery or fetch their own data. This isn't just style — it's so the same component can be rendered in different contexts (Today, House, PlantDetail) without each render duplicating the network request. Data fetching belongs to the page-level component; presentation belongs to the shared one. Mental model: a Rails partial with locals, except the "locals" are typed by JSX and the partial physically can't hit the database because it has no access to the API client.
Don't useMemo trivial expressions. Tempting to wrap arithmetic, string formatting, or small filters in useMemo "for performance". Don't. The memoization bookkeeping (storing the result, comparing deps on every render) costs more than the calculation, and React Compiler is going to fight you on it later. Reach for useMemo only when (a) the work is genuinely expensive — sorting hundreds of items, complex object building — or (b) the result feeds a dependency array that needs reference stability. This is Vercel's rerender-simple-expression-in-memo rule and it's a genuine reversal from advice you may have heard four years ago.
Functions called during render must be deterministic. getVoiceLine(personality, state, plantId) is called from inside a component's JSX. React may re-render that component many times per second during animations — every time, it calls getVoiceLine again. If the function picks a random voice line, the plant's quote will flicker between options on every render. Use quotes[plantId % quotes.length] (or any deterministic seed-to-index mapping) so the output is stable for a given input. The general rule: anything called during render should be pure — same input, same output, no side effects. This is one of those things hobbyist-era React often gets wrong because the bug only surfaces once you start animating.

Forward compatibility notes

No data-fetching inside components. If you find yourself wanting to call usePlant inside HeroCard, stop — pass the plant as a prop instead. Components that fetch their own data are hard to compose, test, and reason about. The screens that consume these (ticket 9+) will do the fetching and pass the data down.
Voice selection must be deterministic. Ticket 9's Today screen may re-render several times per second during animations. If getVoiceLine picks randomly, the plant's voice will flicker between lines on every render. Use quotes[plantId % quotes.length] or similar so the output is stable for a given plant.
Phase 3 will swap PlantAvatar for real illustrations. Keep the prop shape simple (species, size) so a future version can drop in without breaking callers. Don't bake in colour-specific logic that a real illustration won't have.
RoomCard's attentionCount is computed by the House screen, not by RoomCard itself. The House screen will loop over plants per room and tally the ones that need water/feed. RoomCard just renders the number it's given.

Notes & gotchas

  • The ProgressRing SVG math isn't trivial — the plan has a worked version with the correct strokeDasharray/strokeDashoffset calculation. Use it as a reference so you don't spend an hour on circumference math.
  • Framer Motion animations are ticket 13's job. Don't add motion wrappers here.
  • Icons are inline SVG for now. If you want to swap to a library like lucide-react later, fine — but it's not required for this ticket.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch, then open a PR.
