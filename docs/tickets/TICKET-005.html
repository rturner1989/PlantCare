What came before: ticket 3 built the AuthContext with working login(), register(), and logout() methods — they hit the API, store the token, and set the user. Ticket 4 wired /login and /register routes to placeholder pages. This ticket fills those placeholders with real UI.

What this unlocks: ticket 6 (onboarding wizard) starts at /welcome after a successful registration. Without real Login/Register, no one can get into the app — so every remaining ticket needs this one to land first.

Mental model: Rails' SessionsController#new/#create plus a UsersController#new/#create, but the form state lives on the client and the submit hits a JSON endpoint instead of a Rails action.

What you're building

Two pages plus a two-line tweak to App.jsx.

pageclient/src/pages/Login.jsx
Email + password form. On submit, calls useAuth().login(). If login succeeds, navigates to the location the user was originally trying to reach (read from useLocation().state.from) — falling back to / if there isn't one. If it fails, shows the API error message inline. Submit button is disabled while the request is in flight.
pageclient/src/pages/Register.jsx
Name + email + password + password confirmation form. On submit, calls useAuth().register(). On success, navigates to /welcome so the onboarding wizard can take over. On failure, shows API validation errors (duplicate email, password too short, passwords don't match). Same loading/disabled behaviour as Login.
rewriteclient/src/App.jsx
Replace the Login and Register PlaceholderPage elements with the real lazy-imported components. This is the one-line-per-task integration pattern the plan's Task 4 set up — each future screen ticket does the same swap.

Requirements

  1. Login form submits to the auth context. Email + password fields, controlled inputs, submit button. On submit, await login(email, password). On success, navigate to location.state?.from?.pathname if present, else /.
  2. Register form submits to the auth context. Name, email, password, password confirmation. On submit, await register(name, email, password, passwordConfirmation). On success, navigate to /welcome.
  3. API errors surface as a flash toast. Bad credentials, duplicate email, weak password etc. throw from AuthContext's login()/register(); the submit handler catches and shows the error via useToast().error(message). This is the Rails-flash pattern — same channel as every other transient message in the app, rather than a per-form inline slot.
  4. Loading state. While a submission is in flight, the submit button is disabled and shows a loading indicator (spinner or "Signing in…" text). This prevents double-submits.
  5. Cross-linking. Login has a "Need an account? Register" link; Register has a "Already have an account? Log in" link.
  6. Visual matches mockup 06. Logo at top, centred form, mint gradient background, rounded card, big primary button. The spec has specific colour/radius tokens — use those.

Acceptance criteria

Visiting /login shows a real form, not the placeholder page.
Submitting with a wrong password shows an error toast within a second, carrying Rails' real error message.
Submitting a valid login from a ProtectedRoute redirect (e.g. you tried to visit /house while logged out) lands you on /house — not /.
Submitting valid registration lands you on /welcome.
Registering with a mismatched password confirmation shows a "Passwords do not match" error toast — and does so without a server round-trip (check the Network tab: no POST /api/v1/registration should fire).
Registering with an already-taken email shows the server-side "email already taken" error as a toast.
The submit button is visibly disabled during an in-flight request — double-clicking it doesn't fire two requests (check Network tab).
Refreshing the page after login keeps you logged in (session restore from ticket 3 still works).
./scripts/lint.sh and ./scripts/run_tests.sh client pass.

React concepts worth looking up

React Router v7's location state channel. When ProtectedRoute redirected the unauth user to /login, it called <Navigate to="/login" state={{ from: location }} /> — that state object travels with the navigation. Login reads it back via useLocation().state?.from?.pathname to know where to bounce them after auth. Canonical "redirect back to where I was going" pattern in modern React Router. Worth a five-minute docs read if you haven't seen it before — it's the React idiom for what Rails does with store_location + after_sign_in_path_for.
The async-submit lifecycle — and the finally block. Shape of a submit handler in modern React: e.preventDefault() first (otherwise the browser does a full-page form post and the SPA resets), then setSubmitting(true), then try { await login(...) }, then either navigate-on-success or set-error-on-fail, then setSubmitting(false) in a finally block. The finally matters — if you only flip submitting back to false on success, a failed login leaves the button permanently disabled. Easy bug to ship.
Flash-style toasts via ToastContext. Rails has flash[:notice]/flash[:alert] baked into the framework; React has nothing equivalent, so we roll our own. ToastContext exposes toast.success(), toast.error(), toast.warning(), toast.info() — call them from anywhere that has access to the context. Auth errors thrown by login()/register() get caught in the submit handler and pushed into toast.error(err.message). Keeps AuthContext focused on auth state (user + loading) instead of UI state, and gives every screen the same error channel for free.
Disabling the button isn't just polish. Without disabled={submitting}, a fast double-click fires two POST requests in parallel — and you can race yourself into duplicate sessions or stale-state bugs. The disabled+pending pattern prevents it cheaply. Verify in the Network tab: even if you mash the submit button, you should see exactly one request per click.

Forward compatibility notes

Login must read location.state.from. Ticket 4's ProtectedRoute sets this when redirecting unauth users. If you skip reading it, a user trying to visit /house → forced to /login → successful login → lands on / instead of /house. That's a subtle but annoying UX bug.
Register must navigate to /welcome. Ticket 6 (onboarding) assumes any newly-registered user lands at /welcome ready for the 5-step wizard. Don't navigate to / — the onboarding flow wouldn't fire.
Don't pre-build password reset / forgot password. Not in MVP. If the mockup hints at it, leave the link as a visual placeholder.

Notes & gotchas

  • The AuthContext methods already throw on error — wrap the call in try/catch in your submit handler so you can toggle the submitting state correctly in finally.
  • Don't store the password in refs or log it anywhere — it's a controlled input that should only ever exist in state briefly.
  • If submitting and the error from context are both falsy, you can show a subtle hint like "Don't have an account yet?" in the footer. Keep it minimal.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch, then open a PR.
