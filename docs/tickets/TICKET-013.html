TICKET-013

First-run wizard → Today reveal animation

Open Medium

Summary

When a user clicks "Enter your jungle" on Step 5 of onboarding, the jump from /welcome to / snaps instantly — one screen disappears, the next appears, chrome and all. It's the single most celebratory moment in the signup flow and it currently reads as a page navigation.

Build a one-shot entrance animation that plays only on the first Today render after finishing the wizard: the wizard card departs, the main content fades in, the sidebar (desktop) slides in from the left while its nav items stagger-fade, the dock (mobile) slides up from the bottom with the same stagger, and a welcome toast lands. After that one view, the app feels normal forever — every subsequent login is instant, as today.

In one sentence: turn the wizard-finish → Today transition into a single celebratory moment the user sees exactly once per signup.

What came before. TICKET-006 built the full onboarding wizard and wires Welcome.jsx's handleFinish to set localStorage['plantcare_tour_pending'] = 'true' immediately before navigating to /. Nothing currently reads that flag — it's been sitting there waiting for a consumer. This ticket is that consumer. TICKET-008 delivered the Today/House primitives (ProgressRing, PlantAvatar, TaskRow, etc.) and polished the wizard transitions between steps, but still ends the flow with a snap into /.

What this unlocks. TICKET-009 (Today screen proper) will replace the placeholder content with real dashboard widgets — it doesn't need to worry about an entrance animation because that responsibility lives on AppLayout and the chrome components. TICKET-010 (House) benefits from the same one-shot pattern if we later decide to celebrate any first-view milestone (first plant added, first streak). The flag-based gate established here is that pattern, ready to be generalised.

Mental model. Rails' flash[:notice] on after_sign_up_path_for — set once in the controller action that handled registration, consumed on the next page render, gone on the page after that. The plantcare_tour_pending flag is the client-side equivalent: set at the end of the wizard, read once by AppLayout on the next render, cleared immediately so no refresh can replay it.

What you're building

Seven artifacts. One new hook that owns the flag, four components that opt into reveal behaviour when the hook says to, and two small pieces of glue (the welcome toast + reduced-motion fallback).

hookclient/src/hooks/useFirstRunReveal.js
Reads localStorage['plantcare_tour_pending'], clears it on the first read via a useEffect, and returns { isFirstRun: boolean }. Job: answer "should chrome animate in this render?" exactly once per fresh signup — never on reload, never on re-login, never on navigating away and back. All other consumers (Sidebar, Dock, main) read this hook rather than touching localStorage themselves so there's only one place the flag is consumed.
extendclient/src/layouts/AppLayout.jsx
Calls useFirstRunReveal() once at the top, passes firstRun as a prop to Sidebar and Dock, and applies a fadeIn motion wrapper around <main> when the flag is set. Also fires the welcome toast via useEffect (gated on firstRun) so the toast only appears once.
rewriteclient/src/components/Sidebar.jsx
When firstRun, the whole aside slides in from x: -260 to 0 using Framer Motion; once it lands, nav items stagger-fade in at ~80ms intervals. Uses Framer's staggerChildren variant pattern so the orchestration is declarative. When firstRun is false (every normal render), the component renders exactly as it does today — no motion wrappers, no extra DOM.
rewriteclient/src/components/Dock.jsx
The mobile-only counterpart to Sidebar. When firstRun, the dock slides up from the bottom (y: +100 to 0) and the four dock items stagger-fade in. Visible-viewport concern: the dock is lg:hidden, Sidebar is hidden lg:flex, so only one of them actually animates on any given screen.
extendclient/src/layouts/AppLayout.jsx (welcome toast)
On first-run render, fires a single toast via useToast().info() — content along the lines of "Welcome, {firstName} 🌿", reading the user from useAuth(). The effect's dependency array is tight enough that it fires exactly once even if firstRun re-evaluates (it won't, but belt-and-braces).
extendReduced-motion fallback
Every motion wrapper uses useReducedMotion() from motion/react. When the user prefers reduced motion, the reveal collapses to the end state instantly — chrome is there, main is opaque, toast fires — but no slide, no stagger, no fade. Same UX contract, no vestibular trigger.
testclient/tests/pages/first-run-reveal.spec.js
E2E test. Registers + completes the wizard, lands on /, asserts the animation classes / welcome toast are present on the first render. Reloads /, asserts they are absent. Logs out and back in, asserts they stay absent. Verifies localStorage['plantcare_tour_pending'] is cleared after the first view.
App tree during a first-run render: AppLayout ├── useFirstRunReveal() ──► { isFirstRun: true } │ ├── <Sidebar firstRun /> (hidden on mobile) │ ↳ motion.aside x: -260 → 0 │ ↳ motion.nav staggerChildren │ ↳ NavLink × 3 y: 8 → 0, opacity 0 → 1 │ ├── <main> (motion-wrapped when firstRun) │ ↳ motion.div opacity: 0 → 1 │ ↳ <Outlet /> (Today placeholder) │ ├── <Dock firstRun /> (hidden on desktop) │ ↳ motion.nav y: +100 → 0 │ ↳ motion.nav staggerChildren │ ↳ NavLink × 4 y: 8 → 0 │ └── useEffect → toast.info("Welcome, {name} 🌿") After the one-shot: every subsequent render ignores firstRun (the flag is gone from localStorage), so the above collapses to the normal AppLayout tree — no motion wrappers in the DOM.

Requirements

  1. When a user finishes the wizard and lands on / for the first time, they see the sidebar/dock chrome animate in rather than appearing instantly.
  2. The animation plays exactly once per signup. Reloading the page after it's played shows chrome instantly, as it does today. Logging out and back in shows chrome instantly. Navigating from / to another route and back does not replay it.
  3. Users who open the app for the first time via a login (not a signup) never see the animation — only fresh signups do.
  4. On desktop (the lg: breakpoint and up), the sidebar is what slides in. On mobile, the dock is. Both use a stagger-fade on their nav items after landing.
  5. A single welcome toast fires on the first render only, greeting the user by name with a leaf emoji.
  6. Every motion in the sequence respects prefers-reduced-motion — the end state is identical but no slides, fades, or staggers happen when the user opts out.
  7. The total orchestration completes in under ~1 second so the user can start interacting quickly, even though the first interaction is likely just reading the welcome toast.

Acceptance criteria

Register a fresh user → complete the wizard → click "Enter your jungle": sidebar/dock visibly animates into place, main content fades in, welcome toast appears with the user's first name.
After that reveal, refresh the page: no animation, no welcome toast. Just the normal authenticated app.
Log out, log back in with the same credentials: no animation, no welcome toast.
During the reveal, navigate to /discover and back to /: no replay.
In DevTools → Application → Local Storage after the first Today render: plantcare_tour_pending is gone.
System Settings → Accessibility → Reduce Motion = On: chrome still appears, toast still fires, but no slides or fades are visible to the naked eye.
Narrow viewport (<1024px): it's the dock that animates, not the sidebar. Wide viewport: vice-versa. Exactly one chrome element animates per viewport.
./scripts/lint.sh and ./scripts/run_tests.sh client both pass.

React concepts worth looking up

Framer Motion variants + staggerChildren. Variants are named animation states (like "hidden" / "visible") you define once on a parent, and children inherit automatically. Setting transition: { staggerChildren: 0.08 } on the parent means every child that has matching variant names animates 80ms after the previous one — no per-child timing math. This is the clean way to get the nav-items-appearing-one-by-one effect without chaining setTimeouts.
useReducedMotion() from motion/react. Reads the user's prefers-reduced-motion OS setting and returns a boolean. The Toast component already uses it — follow the same pattern: build one set of props for the reduced version (duration: 0 or collapsed initial = animate), and switch on the boolean. Never hard-code an always-on animation; accessibility users get motion sickness from large slides and will leave.
The "localStorage flag that fires once" pattern. Read the flag on mount, decide whether to run the reveal, clear the flag in the same mount's effect. Don't rely on component state alone — React Strict Mode double-mounts effects in dev, so without the clear-on-first-read guard, dev mode would look like the animation is replaying. Think of it as the React equivalent of consuming a Rails flash: show once, clear, never see it again.
useEffect that must fire exactly once — including under Strict Mode double-mount. Dev-mode Strict Mode re-runs effects to catch impurity bugs. A naive useEffect(() => { setFlag(); clearLocalStorage() }, []) will run the "clear" twice and potentially fire the toast twice in dev. Solve by reading + clearing localStorage synchronously in a ref the first time the hook is called, so the second Strict Mode mount sees an empty flag and no-ops. The hook body becomes idempotent — that's what you're aiming for.
Avoid prop-drilling by reading the hook where needed. It's tempting to have AppLayout read firstRun and pass it to Sidebar, Dock, and main as props. Cleaner: call useFirstRunReveal() in each component that needs it. BUT — that breaks the "fires once" guarantee (multiple readers each try to clear the flag). Keep AppLayout as the single reader, pass the value down as props. This is one of the times prop-drilling beats hook proliferation.
Don't animate the chrome's identity — animate its entrance. React Router's route change re-mounts <main>'s children but NOT the surrounding <AppLayout>. That means Sidebar and Dock persist across route changes. Good — we only want their entrance animation on FIRST render of AppLayout, which is exactly when firstRun is true. Subsequent route changes leave them alone. React Router's layout-route pattern is doing the right thing for free.

Notes & gotchas

  • Don't touch Welcome.handleFinish's localStorage write. It already sets the flag correctly. The only surface this ticket cares about is reading it.
  • Strict Mode double-mount is the most likely bug source. In dev, you'll see the reveal play twice if the hook isn't idempotent. Test by saving the flag, hard-reloading /, and confirming the reveal plays once — not once-then-again-250ms-later.
  • Don't call the hook from inside Sidebar and Dock. Only AppLayout calls it. Multiple callers = the "clear on read" race would either clear the flag before some components see it, or replay the animation after some components have already moved on.
  • Don't try to animate the wizard's exit from inside this ticket. The wizard finishes, navigate('/') fires, the wizard unmounts. That's route-level, not component-level — no need to coordinate. AppLayout mounts fresh on the Today route and runs its first-render reveal.
  • Don't introduce a full onboarding tour here. This ticket is just the chrome entrance and welcome toast. Feature tooltips / highlight arrows / dismissible tour cards are a separate scope — park them for a future ticket.
  • Don't assume confetti or sparkle particles. Spec §5.3 mentions a confetti moment for the "all caught up" state — that's a Today-content concern tied to real dashboard data, not this ticket's chrome entrance. Leave room for it, don't pre-empt it.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch (ticket-013), then open a PR.
