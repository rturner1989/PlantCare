Onboarding wizard — first rooms and first plant

Open High

Summary

A five-step wizard at /welcome that newly-registered users land on. By the time they finish, they have at least one room, one species picked, and one plant created — so Today and House aren't empty on first visit.

In one sentence: walk a brand-new user from zero to a populated app in under two minutes.

Step-by-step state machine:

step 1 → "Let's set up your garden" [Continue] │ step 2 → Pick rooms [Create rooms] ──→ Promise.all(POST /rooms) │ │ │←────────────────────────────────────────────────────────────┘ ↓ step 3 → Pick your first species [Continue] (uses GET /species?q=) │ ↑ │ └── search input, debounced via useDeferredValue ↓ step 4 → Pick environment [Create plant] ──→ POST /plants │ (light / temperature / humidity) │ │←────────────────────────────────────────────────────────────┘ ↓ step 5 → "You're all set" [Let's go] ──→ navigate('/')

Requirements

  1. Progress bar. A thin bar at the top of every step showing N / 5 completion. Transitions smoothly between steps.
  2. Step 1 — Welcome. Plant emoji, "Let's set up your garden" headline, short copy, single "Continue" button that advances to step 2.
  3. Step 2 — Rooms. A grid of preset rooms (Living Room, Bedroom, Kitchen, Bathroom, Office, Balcony) the user can toggle on/off. A custom-room input lets them add their own. Selected rooms are listed below. "Create rooms" fires all the POST /rooms requests in parallel via Promise.all and advances on success. Disabled if zero rooms selected.
  4. Step 3 — Species search. Text input, results list. The input feels responsive while typing — no request per keystroke. Clicking a result picks the species and advances. Optional nickname field can be left blank (we default to the species common name).
  5. Step 4 — Environment. Three grouped toggle rows: light (low / medium / bright), temperature (cool / average / warm), humidity (dry / average / humid). Defaults set to medium/average/average. "Create plant" fires POST /plants with the first selected room and advances.
  6. Step 5 — Done. "You're all set" celebration copy, "Let's go" button that navigates to /.
  7. Back button. Every step (except 1 and 5) has a way to go back to the previous step. Back preserves in-progress state (you don't lose your room selections by going back from step 3).
  8. Parallel room creation. When submitting step 2, room POSTs must fire in parallel (Promise.all), not sequentially. This is the Vercel React best-practice async-parallel rule.
  9. Debounced species search. Typing in the search box must not fire a request per keystroke. Use useDeferredValue to hold a "settled" copy of the query and drive the API call from that instead of the raw input.

Acceptance criteria

Registering a new user lands on /welcome step 1.
Clicking "Continue" on step 1 advances to step 2, and the progress bar shows 2/5.
Selecting three rooms and clicking "Create rooms" shows exactly three POST /rooms requests firing in parallel in DevTools Network tab (not one after another).
Typing "mon" fast into the species search does not fire one GET /species per keystroke — you should see at most one or two requests settle on the final value.
Finishing step 5 and clicking "Let's go" navigates to /, and Today shows the rooms and plant you just created.
Clicking "Back" on step 3 returns to step 2 with your previously selected rooms still ticked.
./scripts/lint.sh and ./scripts/run_tests.sh client pass.

React concepts worth looking up

useDeferredValue — new since React 18. Takes a value and returns a "slightly delayed" copy. React keeps the input responsive by letting the raw value update immediately, but defers the downstream work (here, the species API call) until React gets a quiet moment. It's the built-in way to say "this work is non-urgent" without writing a manual setTimeout debounce. Usage: const deferredQuery = useDeferredValue(speciesQuery), then drive your fetch effect from deferredQuery. This is one of the most useful additions in React 18 and post-dates your hobby learning — worth a real read of the docs.
Promise.all for independent work. Fire independent async operations with await Promise.all(items.map(item => apiPost(...))), never a for loop with an await inside. The for-loop version runs sequentially — 5 rooms × 200ms each = 1 second. The Promise.all version runs them in parallel — ~200ms total. This is Vercel's async-parallel rule and it's the single most common React performance bug. (More of a JS rule than a React rule, but Vercel calls it out because it bites React apps constantly.)
Why this ticket uses bare apiPost instead of TanStack Query. Onboarding is a linear, fire-and-forget flow — submit data, advance, done. Nothing to cache, nothing to refetch, nothing to invalidate. TanStack Query earns its keep when multiple components share data over time; here, the rooms and plant you create are written and the user moves on. Use the api/client.js wrapper directly. Reaching for TanStack here would be like using ActiveRecord callbacks for a one-shot insert from a rake task — overkill.
Derive during render, don't sync via useEffect. A common trap from the React 16/17 era: storing a computed value in useState and using useEffect to "keep it in sync" with the source. const progressPercent = (step / totalSteps) * 100 is computed inline on every render — React already re-renders when step changes, so any inline expression is automatically up-to-date. The "useEffect to sync derived state" pattern is now an explicit anti-pattern in the React docs. If something can be derived from props or state, derive it; don't store it.
Effect cleanup with a cancel flag. Your species-search effect fires apiGet when deferredQuery changes. If the user advances to step 4 while a request is still in flight, the response will arrive after the component has moved on — and calling setSpeciesResults at that point either crashes (on older React) or fires a "set state on unmounted component" warning. The fix is the cancel-flag pattern: let cancelled = false; ...; return () => { cancelled = true }, then check if (!cancelled) setState(...) before every set. This is the React idiom for "I started something async; here's how I clean it up if I'm no longer the right component to handle the result."

Forward compatibility notes

Species search pattern becomes ticket 12's starting point. The Add Plant flow in ticket 12 reuses the same search UX. When you build it here, prefer structure that will lift cleanly into a reusable bit later — but don't extract it now. You only know the right shape once you've written the second version.
First plant goes into the first selected room. The plan's logic: step 4 uses createdRooms[0].id as the room_id. That's fine for MVP. A future enhancement could let the user pick which room — don't build that now, but don't hard-code anything that would block it either.
navigate('/', { replace: true }) on finish. Using replace means the browser back button from Today won't bounce the user back into onboarding. Without replace, they'd get sent back through the wizard on every accidental back-swipe.

Notes & gotchas

  • The species search effect needs a cancel flag (let cancelled = false + cleanup function) to handle the case where the user advances past step 3 while a request is still in flight. Without it, you get a "set state on unmounted component" warning.
  • Don't use TanStack Query here just because ticket 7 sets it up. Onboarding is fire-and-forget; direct apiPost/apiGet is the right tool.
  • If the user refreshes mid-wizard, they lose their progress and start at step 1 again. That's acceptable for MVP. Don't try to persist to localStorage.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh client, commit on a feature branch, then open a PR.
