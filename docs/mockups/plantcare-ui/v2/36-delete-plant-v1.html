Design notes

Delete Plant is intentionally the simplest of the CRUD dialogs. The only variant is whether we add a typed-confirmation guard based on plant history. Everything else — shell, layout, copy — stays identical.

  • Destructive CTA convention. Filled --coral-deep → paper text via --gradient-coral + --shadow-cta-danger. The emerald brand gradient is reserved for safe/progressive actions (Adopt, Save, Continue); coral signals "you are about to remove something". Cancel demotes to secondary (--paper-deep fill, ink-soft text) so the eye lands on Delete without it feeling like the default.
  • Why the typed-confirmation guard. High-regret deletes are plants the user has invested in: long history, lots of photos, sentimental nicknames. Gating those behind a name-match forces a 2-second pause — enough to catch "oh wait, wrong Monty". Thresholds (>90 days or 10+ care logs) are illustrative; back-end can tune. Newly-adopted plants skip the guard (Scene A) because the regret cost is low — you probably just typo'd Add Plant.
  • Entry point is Edit Plant. Mockup 34 (Edit Plant) surfaces Delete as a quiet coral text link at the bottom of its footer — not a button. Clicking it dismisses the Edit dialog and opens THIS dialog on top of the same scrim. Cancel here returns the user to the Edit dialog with their unsaved edits intact.
  • Mobile sheet behaviour. Bottom sheet at ~78dvh (shorter than Add/Edit — less content). Cancel moves to a text link top-left in the sheet header; Delete takes the full sticky footer. Drag-down-to-dismiss, scrim-tap, or Cancel all return to Edit Plant without deleting. Dismissal does not require the typed confirmation — only the Delete action does.
  • Photos disclosure. Collapsed by default using native <details>. Answers the most common "wait, what about…" question without bloating the default view. Copy points users at the Journal export path so the hedge-against-regret has a concrete workaround.
  • Open questions. (a) Threshold for typed guard — age vs. log count vs. both, finalise during implementation. (b) Do we keep a server-side soft-delete / 30-day undo regardless of confirmation strength, or is delete truly irreversible? Spec says "can't undo this" — worth confirming with Rob before wiring the backend. (c) Confirmation input — case-sensitive or case-insensitive? Leaning case-sensitive to match the visible label exactly.
