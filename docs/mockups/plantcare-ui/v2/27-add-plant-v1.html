Add Plant locked

The most-used form in the app. FAB is universal; this flow is what it opens. Keep friction low: 3 steps max, smart defaults, optional photo, one primary CTA per step. Same shell is reused for Edit plant (fields pre-filled, CTA changes to "Save"), Log care (single-step variant), and Delete plant (confirmation variant).

  • Shell. Centered dialog on desktop (600×640), Vaul bottom sheet on mobile (~82dvh). Header: eyebrow + Fraunces italic title + close (✕). Progress indicator: 3 dots + label (desktop) / bar + "N / 3" counter (mobile). Body scrolls. Footer: Back left, primary CTA right. Scrim-dim behind, scrim-tap or Escape closes with confirm if form dirty.
  • Step 1 · Species. Search + popular grid + optional photo. Photo is first-class but skippable. Perenual species hydrated on Continue (awaits `/api/v1/species/:id?perenual_id=X` for Perenual-sourced picks to get full care metadata before step 2 calculates schedule).
  • Step 2 · Space + environment. (not mocked) — pick existing space or "+ Add a space" inline. Environment 3 segments (Light / Temperature / Humidity) auto-filled from the space's defaults; override if this plant needs different. "Inherited from Living Room" hint under each segment.
  • Step 3 · Name & commit. Preview card (live-bound to nickname input), schedule calc preview with reasoning, primary "Adopt [Nickname]" CTA.
  • Smart defaults everywhere. Nickname defaults to species common name. Environment inherits from space. First-check-in reminder is auto-scheduled. User ships the form with just 3 taps (species → space → adopt) if they accept defaults.
  • FAB is universal. ＋ button on Today, House, Journal, Encyclopedia, Me — always opens this dialog. Mobile FAB sits in the middle dock slot. Shortcut: N key on desktop.
  • Perenual rate-limit fallback. If Perenual details hit 429 or are missing, Species.build_fallback_from_search persists a minimal row with neutral care defaults (watering 7d, feeding 30d, personality chill). User still completes the flow; schedule is later refined when species hydrates.
  • Reuse for other CRUD. Same shell handles: Edit plant (all fields editable, title "Edit Monty"), Log care (single-step, title "Log watering · Monty"), Delete plant (destructive confirm variant — coral primary CTA "Delete forever"). One dialog primitive + body variants.
  • Relationship to onboarding. Onboarding Step 3 (mockup 19) is a different shell — full-screen wizard, multi-plant batch-add in one session. This dialog is the single-plant post-onboarding flow summoned from the FAB. They share the underlying primitives — SpeciesPicker, PhotoUpload, SpacePicker, EnvSegment, NicknameInput, SchedulePreview — but NOT the shell. Build the primitives once; Onboarding composes them into its wizard pages, Add Plant composes them into this 3-step dialog.
  • Post-adopt celebration. On submit, dialog closes with a scale-down + toast: "Monty joined your greenhouse 🌿". Today's rituals refresh. If first plant, triggers first-run tour instead.
