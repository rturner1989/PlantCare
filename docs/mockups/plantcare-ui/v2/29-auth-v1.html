Auth locked

Three forms, one shell. Desktop split-panel layout with dark-forest marketing on the left, paper form on the right. Mobile collapses to a centred paper card on the mint gradient. Login + Register + Password reset share the exact same field primitive, CTA primitive, social-button primitive, divider. Post-register routes straight into onboarding Step 1 (intent picker, mockup 19) — auth is the doorway, not the destination.

  • Two-panel desktop layout. 50/50 grid. Left: forest-gradient marketing with eyebrow · Fraunces italic headline · short pitch · 3 peek-pills · support tagline. Right: paper form with standard fields + CTA + social sign-in. Mobile ships the form only; pitch compresses into the subtitle.
  • Peek-pills on the marketing panel. Glassmorphic pills over the forest gradient, each showing a concrete hook — "Monty needs watering · 3d overdue" on login (your account has state waiting), "Journal every care action" on register (product value), "Hashed storage · we never see your password" on reset (trust). Never generic.
  • Field primitive. Label (uppercase letter-spaced 10px) + input (paper bg, 1.5px paper-edge border, rounded-md, focus→emerald border + soft glow) + optional hint (Fraunces italic 11px) + optional meta (right-aligned strength chip). Same primitive across all three forms + Me's edit dialog.
  • Password strength meter. `f-meta` positioned top-right inside the password input. Three states: "Weak" (coral), "Fair" (sunshine), "Strong" (emerald). No progress bar — single-word chip keeps it calm. Rule from CLAUDE.md: 12+ chars + mix. Required current-password for password changes (handled in Me · danger zone).
  • Rack::Attack throttle surfaced. Password-reset form's marketing panel names the throttle explicitly — 3 per email per hour, 10 per IP per hour (per PR #31). When user hits the throttle, `RateLimitError` shows as a yellow warning toast (per the `useFormSubmit` convention). Register + Login also Rack-Attack'd when we ship — mention in form copy only if users hit it.
  • Success state = same shell, different body. Post-password-reset-request shows an inline success banner, eyebrow flips to "Check your email", CTA becomes "Open mail app". Don't redirect to a dedicated success page — keep the same URL so back-button works and users can try again with a different email.
  • Social sign-in (Google + Apple). Apple mandatory on iOS if Google is offered (App Store rule). Paper-bg outlined buttons, full-width, below a "or" divider. Phase 2 ship — Phase 1 email/password only, but the design slot is here.
  • Security copy is plain-English. "Hashed · we never see your password" not "bcrypt stretched salted hash". "Valid for one hour · one-time use" not "JWT TTL 3600s, one-time JTI". Users aren't devs. Security confidence comes from clarity, not jargon.
  • Routes into onboarding. Register CTA → onboarding Step 1 (intent picker). Login CTA → first-run tour if first login, else Today. Reset CTA → signs user in automatically after reset (per current refresh-token flow). Auth is a doorway.
  • No voice samples, no personality badges. Auth is pre-account; no plants exist yet. Marketing peek-pills on Login reference your plants (the user has an account with state); on Register they reference the product's value (user has no state). Different pitch, same shape.
